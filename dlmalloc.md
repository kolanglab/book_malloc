# dlmalloc

Doug Lea のアロケータ、通称 **dlmalloc** は、
動的メモリ割り当ての「標準的な教科書実装」である[](#cite:lea2000)。
1987 年から Lea が育て、パブリックドメインで公開されたこのコードは、
glibc malloc（ptmalloc2）の直接の祖先であり、
無数の組み込みシステムや言語処理系に取り込まれてきた。
第 II 部で学んだ部品（境界タグ、bin、分割と併合、指定適合）が
ほぼ理想的な形で組み合わさっているので、
**まず dlmalloc を読むと、後続のすべてが速く読める**。
この章では、その内部構造を部品ごとに分解する。

## チャンクという境界タグの実物

dlmalloc のメモリ単位は[チャンク](#index:チャンク)（chunk）である。
[ヒープを覗く章](inside-malloc.md)で実測した「アドレスが 16 バイト刻み」
「ヘッダが直前 8 バイト」の正体がこれだ。
使用中チャンクと空きチャンクで、同じ領域の意味が変わる。

要求バイト数はそのままチャンクにならない。ヘッダ分を足し、`MALLOC_ALIGNMENT`（64bit では 16）の倍数へ切り上げる（`request2size`）。手元の glibc 2.39 で `malloc_usable_size` を覗くと、丸めの段差がそのまま見える。

```
req  usable  chunk(=usable+8)
  1     24   32
  8     24   32
  9     24   32
 24     24   32
 25     40   48
100    104  112
120    120  128
121    136  144
```

要求 1〜24 バイトはすべて同じ最小チャンク（32 バイト）に丸められ、25 で次の段（48）に上がる。自分が使用中のあいだは直後のチャンクの `prev_size`（フッタ）領域 8 バイトを読む必要がないので、その 8 バイトもユーザに貸せる。だから `usable` はチャンクサイズ−8 まで、つまり要求＋αまで伸びる。この「段差」が次節の bin がサイズぴったりで並ぶ前提になっている。

```
使用中チャンク:                     空きチャンク (small bin):
+---------------------+           +---------------------+
| prev_size (前が空きの時)|        | prev_size           |
| size            |P|M|N|         | size            |P|M|N|
+---------------------+           +---------------------+
| ユーザデータ          |           | fd (次の空きへ)       | ← ここを malloc が返す
| ...                 |           | bk (前の空きへ)       |
|                     |           | (未使用)             |
+---------------------+           +---------------------+
                                  | size (末尾に複製=フッタ)|
                                  +---------------------+
```

`size` フィールドの下位 3 ビットはフラグに使われる
（チャンクは 8/16 バイト境界なので下位ビットは常に 0）。
最重要が **`PREV_INUSE`（P）ビット**で、
「物理的に前のチャンクが使用中か」を表す。
[フリーリストの章](free-list.md)で説明した境界タグの最適化そのものである。
前のチャンクが使用中なら、その `prev_size`（フッタ）を読む必要がないので、
そのスペースをユーザデータに使える。`M` は mmap 由来、
`N` は非メインアリーナを表す（アリーナは glibc 拡張）。

`free` 時、「次の次のチャンクの P ビット（＝次のチャンクが空きか）」と
「自分の P ビット（＝前のチャンクが空きか）」を見て、前後の空きチャンクと併合する。
境界タグ法の理屈が、そのままビット操作になっている。

## サイズ別フリーリストとしての bin

空きチャンクは [bin](#index:bin) と呼ばれるフリーリスト群で管理される。
[サイズクラスの章](segregated-lists.md)の segregated fit の実物だ。
dlmalloc にも版があり、本章で説明するのは
ptmalloc2（glibc）の直接のベースになった **2.7 系**の構成である
（Lea が現在配布する 2.8 系は構成が変わった。後述の NOTE 参照）。
2.7 系の bin は用途別に 4 種類ある。

**fast bin** は、ごく小さいチャンク（既定で最大 160 バイト程度まで）の
単方向リストである。最大の特徴は**併合しない**ことだ。
[サイズクラスの章](segregated-lists.md)で見た quick fit[](#cite:weinstock1988)
そのもので、解放即リスト先頭に push、確保即 pop で速い。
代償として隣接併合の機会を逃すので、ときどき
`malloc_consolidate` でまとめて隣接併合し、いったん unsorted bin へ戻す
（その後の `malloc` が本来の small/large bin へ仕分ける）。

この「先頭に push／先頭から pop」は、解放と逆順に再確保されるという観察可能な振る舞いになる。同じサイズを 3 つ確保→解放→再確保してアドレスを並べてみる。

```c
void *a=malloc(48), *b=malloc(48), *c=malloc(48);
free(a); free(b); free(c);          /* push 順 a,b,c → 先頭は c */
void *x=malloc(48), *y=malloc(48), *z=malloc(48);
/* x==c, y==b, z==a になるはず */
```

手元の glibc 2.39 での実行結果（アドレスは環境により変わる）:

```
確保: a=0x5c2c5b0a52a0 b=0x5c2c5b0a52e0 c=0x5c2c5b0a5320
再確保: x=0x5c2c5b0a5320 y=0x5c2c5b0a52e0 z=0x5c2c5b0a52a0
x==c?1  y==b?1  z==a?1
```

最後に解放した `c` が最初に返る。これが LIFO で、隣接併合をしないからこそ並べ替えなしで成り立つ。

> [!NOTE]
> 厳密には、現在の glibc は 48 バイト程度の小確保をまず **tcache**（[次章](glibc-malloc.md)）に通すため、上の LIFO は実際には tcache が見せている。とはいえ tcache も「解放即 push・確保即 pop の単方向リスト」という fast bin と同じ設計で、ここで観察している原理はまさに dlmalloc の fast bin のものだ。

**small bin** は、一定サイズ以下をサイズぴったりの双方向リストで持つ。
各 bin が単一サイズ（16 バイト刻み）なので、適合は pop だけ。
FIFO（古い空きから再利用）で局所性にも配慮している。

「各 bin が単一サイズ・16 バイト刻み」を具体化すると、要求バイトからチャンクサイズへ丸め（`request2size`）、それを 16 で割って bin 番号にする、という単純な算術になる（64bit、`smallbin_index(sz) = sz>>4`）。

```
req -> chunksize -> smallbin#
  24 ->   32 -> bin 2
  40 ->   48 -> bin 3
  56 ->   64 -> bin 4
 200 ->  208 -> bin 13
 504 ->  512 -> bin 32
```

チャンクサイズが 16 増えるごとに bin 番号がちょうど 1 つ進む。だから small bin の適合は「番号を計算してその bin の先頭を pop するだけ」で、探索が要らない。large bin で 1 つの bin に複数サイズが混ざり best-fit 探索が必要になるのと対照的だ。

**large bin** は、大きいチャンクをサイズ範囲ごとに持つ。
1 つの bin に複数サイズが混ざるので、bin 内はサイズ降順に並べ、
要求には **best-fit**（その bin で要求以上の最小チャンク）で応える。

**unsorted bin** が dlmalloc の巧妙なところだ。
`free` された（fast bin 行きでない）チャンクは、いったんこの
**仕分け前の一時置き場**に入る。次の `malloc` は、まず unsorted bin を
端からなめ、(a) ちょうど合うものがあれば即返し、
(b) 合わなければ適切な small/large bin へ仕分けながら進む。
これにより「解放直後にすぐ同サイズを確保する」典型パターンが、
bin への仕分けコストを払わずに高速化される（遅延仕分け）。

> [!NOTE]
> この 4 bin 構成（fast/small/large/unsorted）は dlmalloc **2.7 系**のもので、
> ほぼそのまま glibc（ptmalloc2）に受け継がれた（[次章](glibc-malloc.md)）。
> 一方、Lea が現在配布する **2.8 系**（`malloc-2.8.6.c`）は構成を整理し直し、
> fast bin と unsorted bin を廃止した。小サイズはサイズぴったりの
> 双方向リスト **smallbin**、大サイズはサイズのビット列で張った
> 2 分トライ **treebin** で best-fit し、さらに直近の分割で余ったチャンクを
> **dv（designated victim）** として優先的に再利用する。
> 今日「dlmalloc のコードを読む」と出会うのはこちらの構成だ。
> fast bin / unsorted bin の系譜を現役で動かしているのは glibc 側である。

## 指定適合という配置方針

dlmalloc の配置方針は、本書が **指定適合**（designated fit）と呼んできた
複数戦略の使い分けである（Lea 自身は全体として best-fit に近いと述べている[](#cite:lea2000)）。

1. まず fast bin / small bin で完全一致を狙う（探索ゼロ）。
2. なければ unsorted bin を一巡（一致すれば即返、ついでに仕分け）。
3. なければ large bin で best-fit。dlmalloc は large bin の探索を
   速くするため、サイズをキーにした**ビットマップ（binmap）**を持ち、
   「空きのある最小の bin」を `ffs` で一発で見つける
   （2.8 系では large 側の bin 自体が前述の treebin になり、
   トライの探索が best-fit を兼ねる）。
4. どの bin も駄目なら、**top チャンク**（ヒープ末尾の未分割領域）から切り出す。
5. top も足りなければ、`brk`/`mmap` で OS から仕入れて top を伸ばす
   （[OS とのインターフェースの章](os-interface.md)）。

この多段構えは、「ほとんどの確保は 1 か 2 で終わる（速い）、
たまの難しい確保だけ 3 以降（正確）」という、
速度と断片化のうまい折衷になっている。
[フリーリストの章](free-list.md)で「best-fit を速く実装するには
サイズをキーにしたデータ構造」と述べたのが、binmap/treebin として実っている。

この binmap+`ffs` の「一発で見つける」は、具体的にはビットマスクと 1 命令の組み合わせだ。bin 5・9・17 に空きがある状態で、bin 7 以上の最小の非空 bin を探す様子を追ってみる。

```c
unsigned binmap = (1u<<5)|(1u<<9)|(1u<<17);  /* = 0x00020220 */
unsigned masked = binmap & (~0u << 7);       /* bin7 未満を落とす */
int idx = __builtin_ffs((int)masked) - 1;    /* 最下位の立ちビット */
```

実行すると:

```
binmap = 0x00020220
need bin>=7 : masked=0x00020200 -> ffs gives bin 9
```

要求未満の bin（ここでは bin 5）をマスクで消し、残りに `ffs`（find first set）をかけると、bin 7 以上で最小の非空 bin = bin 9 が分岐もループもなく得られる。`ffs` は x86 では `bsf`/`tzcnt` 1 命令。「best-fit を速く」のために bin のリストを走査する代わりに、サイズの並びをそのままビット位置へ写したのが binmap だ。

## OS との境界

dlmalloc は小さい確保を `brk` ベースの連続ヒープ（top チャンク）から切り出し、
大きい確保（既定で 128 KiB 以上、動的調整あり＝`M_MMAP_THRESHOLD`）は
個別 `mmap` で満たす。[ヒープを覗く章](inside-malloc.md)の `strace` で
見たとおりの挙動だ。mmap 由来チャンクは `M` ビットで区別され、
`free` で即 `munmap` される。`M_MMAP_THRESHOLD` を動的に上げる
ヒューリスティック（一度大きいのを mmap して解放したら、しきい値を
そのサイズまで上げる）もあり、「大物を繰り返す」負荷で mmap 連発を防ぐ。

## dlmalloc を読む意義

dlmalloc は単一スレッド前提の設計で、現代のマルチコアサーバには
そのままでは向かない（[マルチスレッドの章](multithread.md)の
ロック直列化問題を抱える）。実際、glibc はこれに**アリーナ**と
**tcache** を足して ptmalloc2 にした（[次章](glibc-malloc.md)）。

それでも dlmalloc を最初に読む価値は大きい。
境界タグ・bin・指定適合・top チャンク・mmap しきい値という
**部品の組み合わせ方の「基準形」**がここにあり、
jemalloc も TCMalloc も mimalloc も、この基準形からの
「何を変えたか」として理解できるからだ。
コードも 1 ファイルで、コメントが豊富に付いている。
本書を読んだ今なら、`malloc.c` を頭から追えるはずだ。

> [!TIP]
> 言語処理系を自作していて「とりあえず動く malloc が要る」なら、
> dlmalloc の派生で組み込み向けに整理された **dlmalloc の縮小版**や、
> 後述の TLSF（[リアルタイムの章](realtime.md)）が現実的な選択肢になる。
> フルスクラッチで書くより、これらを読んで改造する方が学びも多い。

## まとめ

dlmalloc は、境界タグによるチャンク管理、
用途別の bin（2.7 系では fast/small/large/unsorted）、
完全一致→unsorted→best-fit→top→OS という指定適合、
そして `brk`/`mmap` の二段調達からなる、
動的メモリ割り当ての「基準実装」である。
第 II 部の部品がほぼ理想形で組み合わさっており、
後続アロケータを読むときの比較基準になる。
次章では、この dlmalloc に並行性対応を足した、
あなたの手元の glibc malloc を見る。
