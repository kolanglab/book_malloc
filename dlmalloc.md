# dlmalloc — すべての原型

Doug Lea のアロケータ、通称 **dlmalloc** は、
動的メモリ割り当ての「標準的な教科書実装」である[](#cite:lea2000)。
1987 年から Lea が育て、パブリックドメインで公開されたこのコードは、
glibc malloc（ptmalloc2）の直接の祖先であり、
無数の組み込みシステムや言語処理系に取り込まれてきた。
第 II 部で学んだ部品——境界タグ、bin、分割と併合、指定適合——が
ほぼ理想的な形で組み合わさっているので、
**まず dlmalloc を読むと、後続のすべてが速く読める**。
この章では、その内部構造を部品ごとに分解する。

## チャンク — 境界タグの実物

dlmalloc のメモリ単位は[チャンク](#index:チャンク)（chunk）である。
[ヒープを覗く章](inside-malloc.md)で実測した「アドレスが 16 バイト刻み」
「ヘッダが直前 8 バイト」の正体がこれだ。
使用中チャンクと空きチャンクで、同じ領域の意味が変わる。

```
使用中チャンク:                     空きチャンク (small bin):
+---------------------+           +---------------------+
| prev_size (前が空きの時)|        | prev_size           |
| size            |P|M|N|         | size            |P|M|N|
+---------------------+           +---------------------+
| ユーザデータ          |           | fd (次の空きへ)       | ← ここを malloc が返す
| ...                 |           | bk (前の空きへ)       |
|                     |           | (未使用)             |
+---------------------+           +---------------------+
                                  | size (末尾に複製=フッタ)|
                                  +---------------------+
```

`size` フィールドの下位 3 ビットはフラグに使われる
（チャンクは 8/16 バイト境界なので下位ビットは常に 0）。
最重要が **`PREV_INUSE`（P）ビット**で、
「**物理的に前の**チャンクが使用中か」を表す。
[フリーリストの章](free-list.md)で説明した境界タグの最適化そのものだ：
前のチャンクが使用中なら、その `prev_size`（フッタ）を読む必要がないので、
そのスペースをユーザデータに使える。`M` は mmap 由来、
`N` は非メインアリーナを表す（アリーナは glibc 拡張）。

`free` 時、このチャンクの「次のチャンクの P ビット」と
「自分の P ビット（前が空きか）」を見て、前後の空きチャンクと併合する。
境界タグ法の理屈が、そのままビット操作になっている。

## bin — サイズ別フリーリストの集合体

空きチャンクは [bin](#index:bin) と呼ばれるフリーリスト群で管理される。
[サイズクラスの章](segregated-lists.md)の segregated fit の実物だ。
dlmalloc/glibc は bin を用途別に 4 種類持つ。

**fast bin** は、ごく小さいチャンク（既定で最大 160 バイト程度まで）の
単方向リストである。最大の特徴は**併合しない**こと
——[サイズクラスの章](segregated-lists.md)で見た quick fit[](#cite:weinstock1988)
そのもので、解放即リスト先頭に push、確保即 pop で速い。
代償として隣接併合の機会を逃すので、ときどき
`malloc_consolidate` でまとめて本来の bin へ移して併合する。

**small bin** は、一定サイズ以下を**サイズぴったりの**双方向リストで持つ。
各 bin が単一サイズ（16 バイト刻み）なので、適合は pop だけ。
FIFO（古い空きから再利用）で局所性にも配慮している。

**large bin** は、大きいチャンクをサイズ範囲ごとに持つ。
1 つの bin に複数サイズが混ざるので、bin 内はサイズ降順に並べ、
要求には **best-fit**（その bin で要求以上の最小チャンク）で応える。

**unsorted bin** が dlmalloc の巧妙なところだ。
`free` された（fast bin 行きでない）チャンクは、いったんこの
**仕分け前の一時置き場**に入る。次の `malloc` は、まず unsorted bin を
端からなめ、(a) ちょうど合うものがあれば即返し、
(b) 合わなければ適切な small/large bin へ仕分けながら進む。
これにより「解放直後にすぐ同サイズを確保する」典型パターンが、
bin への仕分けコストを払わずに高速化される（遅延仕分け）。

## 配置方針 — 「指定適合」

dlmalloc の配置方針は、Lea 自身が **指定適合**（designated fit）と呼ぶ、
複数戦略の使い分けである[](#cite:lea2000)。

1. まず fast bin / small bin で**完全一致**を狙う（探索ゼロ）。
2. なければ unsorted bin を一巡（一致すれば即返、ついでに仕分け）。
3. なければ large bin で **best-fit**。dlmalloc は large bin の探索を
   速くするため、サイズをキーにした**ビットマップ（binmap）**を持ち、
   「空きのある最小の bin」を `ffs` で一発で見つける。
   （glibc の large bin、より大きいチャンク向けの実装では、
   サイズで張った 2 分トライ＝treebin を使う変種もある。）
4. どの bin も駄目なら、**top チャンク**（ヒープ末尾の未分割領域）から切り出す。
5. top も足りなければ、`brk`/`mmap` で OS から仕入れて top を伸ばす
   （[OS とのインターフェースの章](os-interface.md)）。

この多段構えは、「ほとんどの確保は 1 か 2 で終わる（速い）、
たまの難しい確保だけ 3 以降（正確）」という、
速度と断片化のうまい折衷になっている。
[フリーリストの章](free-list.md)で「best-fit を速く実装するには
サイズをキーにしたデータ構造」と述べたのが、binmap/treebin として実っている。

## OS との境界

dlmalloc は小さい確保を `brk` ベースの連続ヒープ（top チャンク）から切り出し、
大きい確保（既定で 128 KiB 以上、動的調整あり＝`M_MMAP_THRESHOLD`）は
個別 `mmap` で満たす。[ヒープを覗く章](inside-malloc.md)の `strace` で
見たとおりの挙動だ。mmap 由来チャンクは `M` ビットで区別され、
`free` で即 `munmap` される。`M_MMAP_THRESHOLD` を動的に上げる
ヒューリスティック（一度大きいのを mmap して解放したら、しきい値を
そのサイズまで上げる）もあり、「大物を繰り返す」負荷で mmap 連発を防ぐ。

## dlmalloc を読む意義

dlmalloc は単一スレッド前提の設計で、現代のマルチコアサーバには
そのままでは向かない（[マルチスレッドの章](multithread.md)の
ロック直列化問題を抱える）。実際、glibc はこれに**アリーナ**と
**tcache** を足して ptmalloc2 にした（[次章](glibc-malloc.md)）。

それでも dlmalloc を最初に読む価値は大きい。
境界タグ・bin・指定適合・top チャンク・mmap しきい値という
**部品の組み合わせ方の「基準形」**がここにあり、
jemalloc も TCMalloc も mimalloc も、この基準形からの
「何を変えたか」として理解できるからだ。
コードも 1 ファイルで、コメントが豊富に付いている。
本書を読んだ今なら、`malloc.c` を頭から追えるはずだ。

> [!TIP]
> 言語処理系を自作していて「とりあえず動く malloc が要る」なら、
> dlmalloc の派生で組み込み向けに整理された **dlmalloc の縮小版**や、
> 後述の TLSF（[リアルタイムの章](realtime.md)）が現実的な選択肢になる。
> フルスクラッチで書くより、これらを読んで改造する方が学びも多い。

## まとめ

dlmalloc は、境界タグによるチャンク管理、
用途別の bin（fast/small/large/unsorted）、
完全一致→unsorted→best-fit→top→OS という指定適合、
そして `brk`/`mmap` の二段調達からなる、
動的メモリ割り当ての「基準実装」である。
第 II 部の部品がほぼ理想形で組み合わさっており、
後続アロケータを読むときの比較基準になる。
次章では、この dlmalloc に並行性対応を足した、
あなたの手元の glibc malloc を見る。
