# ヒープのセキュリティ

C/C++ のメモリ安全性バグの多くは、ヒープで起きる。
バッファオーバーフロー、解放後利用（use-after-free）、二重解放——
これらは単なるクラッシュではなく、攻撃者がプログラムを
乗っ取る足がかりになる。なぜなら、アロケータの管理情報
（フリーリストのポインタやチャンクヘッダ）がヒープ上に
ユーザデータと混在しており、それを書き換えられると
**アロケータ自身を操作できる**からだ。
この章では、ヒープ攻撃の仕組みを理解し、
それに対抗する確率的・決定的な防御アロケータの研究を見る。
攻撃手法は防御を設計・評価するために学ぶのであって、悪用のためではない。

## なぜヒープが狙われるのか — 管理情報との同居

[フリーリストの章](free-list.md)で見たように、
古典的なアロケータは空きブロックの**本体**に next/prev ポインタを置く。
glibc malloc も、空きチャンクのユーザ領域だった場所に
bin の双方向リンクを格納する。つまり、

- **使用中**のブロックには、直前にチャンクサイズ等のヘッダがある。
- **解放済み**のブロックには、その中にフリーリストのポインタがある。

この同居が、攻撃の温床になる。代表的な手口を 2 つ見よう。

ひとつは**ヒープオーバーフロー**である。
あるブロックの末尾を越えて書き込むと、**隣のブロックのヘッダ**を上書きできる。
隣が空きチャンクなら、そのリンクポインタを書き換えて、
「次に確保されるアドレス」を攻撃者の狙った場所にずらせる
（古典的な unlink 攻撃や、glibc の tcache poisoning がこの系統だ）。
やがてその偽アドレスが `malloc` の戻り値として返り、
攻撃者は任意のアドレスに書き込む足場を得る。

もうひとつが**解放後利用**（[use-after-free](#index:use-after-free)）である。
`free` した後もポインタ（dangling pointer; 宙ぶらりんのポインタ）を
使い続けると、その領域は**別の用途で再確保**されているかもしれない。
攻撃者は、解放と再確保の間に自分のデータを同じ場所へ確保させ
（heap spraying / heap grooming）、被害者が古いポインタ経由で
それを「別の型のオブジェクト」として扱うよう仕向ける。
[API の章](api-basics.md)で「`free` 後に `NULL` を代入せよ」と言ったのは、
この dangling pointer を消すためだった。

> [!CAUTION]
> これらの攻撃は「確保パターンを攻撃者がある程度操れる」ことを前提とする。
> ブラウザや言語処理系のように、外部入力でオブジェクトを大量生成できる環境は
> まさにそれにあたり、JavaScript エンジンや画像パーサの
> use-after-free が実際の攻撃に多用されてきた。
> アロケータの挙動が**決定的**（同じ操作列で必ず同じ配置）だと、
> 攻撃者は配置を予測しやすくなる。次に見る防御は、この予測可能性を崩す。

## 確率的防御 — DieHard と DieHarder

Berger と Zorn の **DieHard**[](#cite:berger2006) は、
発想を逆転させた防御アロケータである。
「攻撃しにくくする」ために、**わざとランダムに配置する**。

- ヒープを必要量より大きく取り（たとえば 2 倍）、
  オブジェクトを**ランダムな空きスロット**に置く。
- スロットがまばらなので、オーバーフローが隣の生きたオブジェクトに
  当たる確率が下がる。当たっても位置が予測できない。
- 同じプログラムを複数の異なる乱数で**並走**させ、出力が食い違ったら
  メモリ破壊を検知する、という冗長化モードもある。

DieHard は元々**信頼性**（確率的メモリ安全性）のための設計だったが、
続く **DieHarder**[](#cite:novark2010) は明確に**セキュリティ**へ舵を切った。
攻撃者の視点で「どれだけ配置を予測されにくくできるか」を解析し、
ヒープの予測可能性を最小化する設計を導いた。
スロットの完全分離、ヘッダのアウトオブライン化（管理情報をデータと分ける）、
確保アドレスのランダム化などである。

この「ヘッダをデータから分離する」「ランダム化する」という二大原則は、
後続の実用的セキュリティアロケータにそのまま受け継がれた。
**FreeGuard**[](#cite:silvestro2017) は、これらの防御を
**ほぼ無視できる速度低下**で実現することを目標に、
サイズクラスとシャドウメモリを使って高速化した。
続く **Guarder**[](#cite:silvestro2018) は、
防御の強度（ランダム化のエントロピー）を**実行時に調整できる**ようにし、
速度と安全性のトレードオフをアプリごとに選べるようにした。

## 解放後利用を断つ — 一度きりのアドレス

確率的防御は「当てにくくする」が、保証はしない。
**use-after-free を原理的に不可能にする**アプローチも研究されている。

ひとつの道は、**アドレスを二度と再利用しない**ことだ。
**FFmalloc**（Fast Forward Allocation）[](#cite:wickman2021) は、
`free` されたアドレスを**決して再確保に使わない**。
仮想アドレス空間を一方向に消費し続ける（広大な 64 ビット空間だからできる）。
こうすると、dangling pointer が指す先には二度と別オブジェクトが来ないので、
use-after-free が「別オブジェクトの乗っ取り」に化けることがない。
物理メモリは `madvise`（[OS とのインターフェースの章](os-interface.md)）で返すので、
仮想アドレスは使い捨てても物理メモリは枯渇しない。

もうひとつの道は、**まだ誰かが指しているメモリは解放しない**ことだ。
**MarkUs**[](#cite:ainsworth2020) は、`free` を即座に実行せず
「隔離（quarantine）」に置き、ヒープとレジスタ・スタックを走査して
**そのブロックを指すポインタがもうどこにもない**と確認できてから
初めて本当に解放する。これは保守的 GC のポインタ走査を
「安全な解放」のために流用した設計で、
use-after-free を低オーバーヘッドで防ぐ。

## ハードウェアで型と寿命を守る — CHERI

ソフトウェアの工夫の限界を、ハードウェアで突破しようとする流れもある。
**CHERI**（Capability Hardware Enhanced RISC Instructions）は、
ポインタを単なるアドレスではなく**ケイパビリティ**——
「このアドレスからこの範囲まで、この権限で」という、
ハードウェアが検証するトークン——に置き換えるアーキテクチャである。
範囲外アクセスはハードウェアが弾くので、ヒープオーバーフローが原理的に止まる。

残る課題は時間的安全性（use-after-free）だった。
**Cornucopia**[](#cite:filardo2020) は、CHERI 上で
「解放されたメモリを指すケイパビリティをすべて無効化する」
スイープ機構を提案し、ヒープの時間的安全性をハードウェア支援で実現した。
ソフトウェアだけでは重い「全ポインタの追跡」を、
ケイパビリティのタグビットを使って現実的なコストに収めている。

## 本番環境で見つける — サンプリング検知

防御で「起きないようにする」のとは別に、
本番環境でバグを**見つけて直す**アプローチも重要だ。
[ツールの付録](tools.md)で扱う AddressSanitizer[](#cite:serebryany2012) は
強力だが、2 倍前後の速度低下があり常時本番投入には重い。

そこで Google の **GWP-ASan**[](#cite:serebryany2023) は、
**ごく低い確率でサンプリング**して、選ばれた確保だけを
ガードページ付きの特別なページに置く。普段はほぼゼロコストで動き、
たまたまサンプルされたブロックでオーバーフローや use-after-free が
起きるとその場で捕まえる。1 台では当たりにくいが、
**数百万台のフリート全体で薄く広く張れば**、本番でしか出ない
メモリバグを継続的に拾える。確率的防御と同じ「数で勝つ」発想が、
今度は検知に使われているわけだ。

## 言語処理系から見たヒープ安全性

最後に、言語処理系の視点を述べておく。
Ruby や Python のような**メモリ安全な言語**は、
そもそもユーザに生ポインタを渡さないので、ここで述べた攻撃は
**処理系自身（C で書かれた部分）と C 拡張**にしか効かない。
逆に言えば、処理系の C コードの use-after-free は、
スクリプト側からは触れないはずのメモリ安全性を破る重大バグになる。
だから CRuby の開発では AddressSanitizer ビルドでのテストが日常的に行われ、
GC との相互作用（解放したオブジェクトを GC がいつ回収するか）にも
慎重な設計がなされている。
処理系を書くということは、ユーザに安全を提供するために、
自分は危険な C のヒープと正面から向き合うということでもある。

## まとめ

ヒープが攻撃されるのは、アロケータの管理情報がデータと同居し、
配置が予測可能だからである。防御の系譜は、
確率的に当てにくくする（DieHard→DieHarder→FreeGuard→Guarder）、
アドレスを使い捨てて use-after-free を断つ（FFmalloc）、
指されている間は解放しない（MarkUs）、
ハードウェアでケイパビリティとして守る（CHERI/Cornucopia）、
本番でサンプリング検知する（GWP-ASan）と多岐にわたる。
いずれも「速度・メモリ・安全性」の三角形のどこかを選ぶ設計である。
次章では、この三角形に**時間の予測可能性**という第四の軸が加わる
リアルタイムシステムのアロケータを見る。
