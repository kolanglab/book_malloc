# リアルタイムと組み込み — 最悪実行時間の保証

ここまでの章は「平均的に速い・無駄が少ない」を目指してきた。
だが、ある種のシステムでは**平均**ではなく**最悪**がすべてだ。
エアバッグの制御、ロボットの関節制御、宇宙機の姿勢制御——
これらは「だいたい速い」では困る。「**必ず** N マイクロ秒以内に終わる」
という保証が要る。この章では、最悪実行時間（WCET; Worst-Case Execution Time）を
保証する動的メモリ割り当て、とりわけ TLSF アルゴリズムを学び、
リアルタイム・組み込みの世界でのメモリ管理の作法を見る。

## なぜ普通の malloc ではいけないのか

リアルタイムシステムが満たすべき性質は、まず **O(1) の上界**である。
これまで見てきたアロケータの多くは、平均は速くても最悪は速くない。

- first-fit/best-fit の線形探索（[フリーリストの章](free-list.md)）は、
  空き断片が増えれば探索が伸びる。最悪 $O(n)$。
- 即時併合は連鎖しうるし、glibc の `malloc_consolidate` のような
  「ためてまとめて処理」は、その回だけ突発的に長くかかる。
- ヒュージページの purge やアリーナのロック待ち（[マルチスレッドの章](multithread.md)）も、
  最悪時間を予測不能にする。

「ほとんどの呼び出しは 50 ns だが、1000 回に 1 回 50 µs かかる」
というアロケータは、平均性能は素晴らしくてもリアルタイムには使えない。
**最悪が有界で、しかも小さい**ことが要件なのだ。

加えて、リアルタイム・組み込みでは断片化も致命的になりうる。
何週間も再起動せずに動き続けるシステムでは、
じわじわ進む外部断片化がいつか確保失敗を引き起こす。
**時間の予測可能性**と**断片化の有界性**の両立が求められる。

## TLSF — 二段ビットマップで O(1)

この要件に応えた代表的アルゴリズムが、Masmano らの
**TLSF**（Two-Level Segregated Fit）[](#cite:masmano2004) である。
名前のとおり、[サイズクラスの章](segregated-lists.md)の segregated fit を
**二段**にして、サイズからフリーリストを**ビット演算だけ**で
引けるようにしたものだ。確保・解放・併合がすべて真の $O(1)$ になる。

仕組みはこうだ。サイズを「2 の冪の区間（第一レベル）」と
「その区間を線形に N 分割した小区間（第二レベル）」の組で表す。

- **第一レベル**: サイズの最上位ビット位置。`fls(size)`（最上位の 1 の位置）で
  即座に求まる。これが「$2^k$ 〜 $2^{k+1}$ のどの区間か」を決める。
- **第二レベル**: その区間内をさらに $2^L$ 等分したどこか。
  上位ビットの続きを取り出すだけ。

各 (第一, 第二) の組にフリーリストがあり、
「使えるリストがある区間」をビットマップで持つ。
要求サイズ以上の最小の空きを探すのは、
**ビットマップに対する `ffs`（最下位の 1 を探す命令）**一発で済む。
ループも探索もない。これが「二段ビットマップで O(1)」の核心である。

```ruby
# TLSF のサイズ→(第一, 第二) 写像の核心部分を Ruby で（SLI = 第二レベルのビット数）
def mapping(size, sli: 4)
  fl = size.bit_length - 1               # 第一レベル = 最上位ビット位置 (fls)
  sl = (size >> (fl - sli)) - (1 << sli) # 第二レベル = 続く sli ビット
  [fl, sl]
end

p mapping(100)   # => [6, 9]    64〜128 の区間を 16 分割した 9 番目
p mapping(128)   # => [7, 0]    ちょうど 128 は次の区間の先頭
p mapping(200)   # => [7, 9]
```

第一レベルが `bit_length - 1`、第二レベルがその下の数ビット、という
**純粋なビット演算**で添字が出る。確保時は「この添字以上で空きのある
最小のリスト」をビットマップで探し、解放時は併合して
対応するリストへ繋ぎ直す——どれも定数回の操作だ。
分割の余りや併合の有界性も解析されており、断片化の上界も保証される。

TLSF は組み込み Linux、RTOS、各種ミドルウェアで広く採用され、
オープンソース実装も複数ある。
「O(1)・低断片化・小さなコード」という三拍子が、
資源の限られた組み込み環境に合致したためだ。

## 「そもそも動的確保を禁じる」という流儀

リアルタイム・安全クリティカルの世界には、
もっと過激な——そして実務的には主流の——立場がある。
**実行中の動的メモリ割り当てを禁止する**、というものだ。

航空・宇宙・自動車の安全規格に強い影響を与えた
MISRA C や、NASA/JPL のコーディング規約は、
「初期化フェーズが終わったら `malloc`/`free` を呼ばない」ことを推奨・要求する。
理由はこれまでの裏返しだ。動的確保があると、
**最悪実行時間も、メモリ枯渇の有無も、静的に証明しにくくなる**。
確保が失敗しうること自体が、検証の難しい分岐を増やす。

その代わりに使われるのが、

- **静的確保**: 必要なバッファを全部コンパイル時／起動時に確保し、以後は使い回す。
- **メモリプール / オブジェクトプール**: 固定サイズ・固定個数の枠を
  起動時に確保しておき、`malloc`/`free` の代わりに枠の貸し借りをする。
  これは本質的に[スラブ](buddy-slab.md)を静的にしたもので、
  確保は枠リストからの pop（O(1)・失敗は「枠切れ」のみ）になる。
- **リージョン / アリーナ**: フェーズの終わりに一括解放する。

TLSF のような「動的だが有界」なアロケータは、
これらが使えない（本当に実行時までサイズが読めない）場面の最後の手段、
という位置づけになる。

> [!NOTE]
> ガベージコレクションを持つ言語にも「リアルタイム GC」の研究があり、
> GC の停止時間を有界化する技術が積み上げられてきた。
> 本書の範囲外だが、動機は本章とまったく同じ——
> 「メモリ管理の最悪時間を有界にする」——である。
> malloc 側の TLSF と GC 側のリアルタイム GC は、同じ問いへの別解だ。

## 組み込みでの実践的な勘どころ

最後に、組み込みでアロケータと付き合うときの実務的な注意を挙げておく。

第一に、**確保失敗を必ず処理する**。デスクトップでは
[オーバーコミット](os-interface.md)のせいで `malloc` が `NULL` を
返さないことも多いが、組み込みは物理メモリが小さく
オーバーコミットもないため、`NULL` は普通に返る。
返り値検査を省くと、本当に落ちる。

第二に、**メモリプールをサイズ／用途ごとに分ける**。
1 つの汎用ヒープに全部を混ぜると、ある用途の確保が
別の用途の断片化を引き起こし、問題の切り分けが難しくなる。
用途別プールなら「どのプールが枯渇したか」が一目で分かり、
最悪量も用途ごとに見積もれる。

第三に、**最悪を測る**。平均レイテンシではなく、
最悪レイテンシ（ヒストグラムの裾）と、長時間稼働後の
断片化（high-water の伸び）を測定する。
[断片化の章](fragmentation.md)で作った「high_water とピーク生存量の比」を
長時間流して、じわじわ伸びていないかを確認するのが定石だ。

## まとめ

リアルタイム・組み込みでは、平均ではなく**最悪**が要件になる。
TLSF[](#cite:masmano2004) は二段の segregated fit を
ビットマップで引くことで、確保・解放・併合すべてを真の $O(1)$ にし、
断片化の上界も保証する。一方、安全クリティカルの主流は
「実行中の動的確保を禁じ、静的確保やメモリプールで済ませる」流儀であり、
TLSF はそれが使えない場面の有界な動的解である。
共通する問いは「メモリ管理の最悪を有界にできるか」で、
これはリアルタイム GC とも地続きだ。
次章では、視点を現在から未来へ——機械学習・ハードウェア・
コンパクションといった最新研究へ移す。
