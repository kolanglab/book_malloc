# リアルタイムと組み込みの最悪実行時間保証

ここまでの章は「平均的に速い、無駄が少ない」を目指してきた。
だが、ある種のシステムでは平均ではなく**最悪**がすべてだ。
エアバッグの制御、ロボットの関節制御、宇宙機の姿勢制御は、
「だいたい速い」では困る。
「必ず N マイクロ秒以内に終わる」という保証が要る。
この章では、最悪実行時間（WCET; Worst-Case Execution Time）を
保証する動的メモリ割り当て、とりわけ TLSF アルゴリズムを学び、
リアルタイム・組み込みの世界でのメモリ管理の作法を見る。

## なぜ普通の malloc ではいけないのか

リアルタイムシステムが満たすべき性質は、まず **O(1) の上界**である。
これまで見てきたアロケータの多くは、平均は速くても最悪は速くない。

- first-fit/best-fit の線形探索（[フリーリストの章](free-list.md)）は、
  空き断片が増えれば探索が伸びる。最悪 $O(n)$。
- 即時併合は連鎖しうるし、glibc の `malloc_consolidate` のような
  「ためてまとめて処理」は、その回だけ突発的に長くかかる。
- ヒュージページの purge やアリーナのロック待ち（[マルチスレッドの章](multithread.md)）も、
  最悪時間を予測不能にする。

「ほとんどの呼び出しは 50 ns だが、1000 回に 1 回 50 µs かかる」
というアロケータは、平均性能は素晴らしくてもリアルタイムには使えない。
最悪が有界で、しかも小さいことが要件なのだ。

この「裾」は手元の glibc でも簡単に観察できる。`malloc(48)` を 10 万回呼んで各回の所要時間を測り、分位点を見てみよう。

```c
// malloc(48) を N 回、各回のレイテンシ(ns)を測ってソートし分位点を出す
for (int i = 0; i < N; i++) { long t0 = ns(); p[i] = malloc(48); d[i] = ns() - t0; }
qsort(d, N, sizeof(long), cmp);
// p50 = d[N/2], p99 = d[N*99/100], p99.9 = d[N*999/1000], max = d[N-1]
```

```
malloc(48) 100000 回, ナノ秒:
  p50  = 1397
  p99  = 2864
  p99.9= 9079
  max  = 29055  (p50 の 20 倍)
```

値は環境により変わるが、傾向は安定している。p50 と p99 は約 1.4 µs と約 2.9 µs で同じオーダーに収まるのに、p99.9 で約 6 倍、最悪では十数〜数十倍に跳ねる（`max` は実行ごとに大きくぶれる）。「中央付近は素晴らしいが裾が長い」を地で行く分布だ。リアルタイムで効くのはこの `max` の側であり、それが有界だと示せないアロケータは使えない。

加えて、リアルタイム・組み込みでは断片化も致命的になりうる。
何週間も再起動せずに動き続けるシステムでは、
じわじわ進む外部断片化がいつか確保失敗を引き起こす。
時間の予測可能性と断片化の有界性の両立が求められる。

## 二段ビットマップで O(1) にする TLSF

この要件に応えた代表的アルゴリズムが、Masmano らの
**TLSF**（Two-Level Segregated Fit）[](#cite:masmano2004) である。
名前のとおり、[サイズクラスの章](segregated-lists.md)の segregated fit を
二段にして、サイズからフリーリストをビット演算だけで
引けるようにしたものだ。確保と解放と併合がすべて真の $O(1)$ になる。

仕組みはこうだ。サイズを「2 の冪の区間（第一レベル）」と
「その区間を線形に N 分割した小区間（第二レベル）」の組で表す。

- **第一レベル**: サイズの最上位ビット位置。`fls(size)`（最上位の 1 の位置）で
  即座に求まる。これが「$2^k$ 〜 $2^{k+1}$ のどの区間か」を決める。
- **第二レベル**: その区間内をさらに $2^L$ 等分したどこか。
  上位ビットの続きを取り出すだけ。

各 (第一, 第二) の組にフリーリストがあり、
「使えるリストがある区間」をビットマップで持つ。
要求サイズ以上の最小の空きを探すのは、
**ビットマップに対する `ffs`（最下位の 1 を探す命令）**一発で済む。
ループも探索もない。これが「二段ビットマップで O(1)」の核心である。

探索が本当にビット演算だけで終わる様子を、具体的な数値で追ってみよう。いま空きブロックがサイズ 80・130・300・5000 の 4 つだけあるとする。`mapping`（SLI=4）で添字に直すと、それぞれ (6,4)・(7,0)・(8,2)・(12,3) のバケットに入る。ビットマップにはこの 4 ビットだけが立つ。

ここに 200 バイトの要求が来たとする。`mapping(200)` は (7,9)。探索はこうだ。

1. 第一レベル 7 の中で第二レベル 9 以上の空きを探す。`sl_bitmap[7]` は bit0（130）だけ。`sl_bitmap[7] & (~0 << 9)` は 0 になり、同じ区間に手頃な空きはない。
2. ならば第一レベル 8 以上で最小の空きを探す。`fl_bitmap & (~0 << 8)` の最下位ビットを `ffs` で取ると 8（300）。その中の最小第二レベルを `ffs` で取って (8,2)。

どちらも `&` と `ffs` だけで、ループなしに「要求 200 を満たす最小の空き＝300 のバケット」へ一発で着く。実際に走らせると：

```
request=   64 -> bucket=[6, 4]
request=  100 -> bucket=[7, 0]
request=  130 -> bucket=[7, 0]
request=  200 -> bucket=[8, 2]
request= 4096 -> bucket=[12, 3]
```

要求 100 が (7,0) に飛ぶ（同区間に空きがなく一段上の 130 へ）のも、200 が 130 を飛ばして 300 を選ぶのも、すべてビットマスクの結果だ。空き集合がどれだけ散らばっていても操作回数は変わらない。これが $O(1)$ の正体である。

```ruby
# TLSF のサイズ→(第一, 第二) 写像の核心部分を Ruby で（SLI = 第二レベルのビット数）
def mapping(size, sli: 4)
  fl = size.bit_length - 1               # 第一レベル = 最上位ビット位置 (fls)
  sl = (size >> (fl - sli)) - (1 << sli) # 第二レベル = 続く sli ビット
  [fl, sl]
end

p mapping(100)   # => [6, 9]    64〜128 の区間を 16 分割した 9 番目
p mapping(128)   # => [7, 0]    ちょうど 128 は次の区間の先頭
p mapping(200)   # => [7, 9]
```

第一レベルが `bit_length - 1`、第二レベルがその下の数ビット、という
純粋なビット演算で添字が出る。確保時は「この添字以上で空きのある
最小のリスト」をビットマップで探し、解放時は併合して
対応するリストへ繋ぎ直す。どれも定数回の操作だ。
分割の余りや併合の有界性も解析されており、断片化の上界も保証される。

TLSF は組み込み Linux、RTOS、各種ミドルウェアで広く採用され、
オープンソース実装も複数ある。
「O(1)、低断片化、小さなコード」という三拍子が、
資源の限られた組み込み環境に合致したためだ。

## 「そもそも動的確保を禁じる」という流儀

リアルタイム・安全クリティカルの世界には、
もっと過激な（そして実務的には主流の）立場がある。
**実行中の動的メモリ割り当てを禁止する**、というものだ。

航空、宇宙、自動車の安全規格に強い影響を与えた
MISRA C や、NASA/JPL のコーディング規約は、
「初期化フェーズが終わったら `malloc`/`free` を呼ばない」ことを推奨・要求する。
理由はこれまでの裏返しだ。動的確保があると、
最悪実行時間も、メモリ枯渇の有無も、静的に証明しにくくなる。
確保が失敗しうること自体が、検証の難しい分岐を増やす。

その代わりに使われるのが、

- **静的確保**: 必要なバッファを全部コンパイル時／起動時に確保し、以後は使い回す。
- **メモリプール / オブジェクトプール**: 固定サイズ・固定個数の枠を
  起動時に確保しておき、`malloc`/`free` の代わりに枠の貸し借りをする。
  これは本質的に[スラブ](buddy-slab.md)を静的にしたもので、
  確保は枠リストからの pop（O(1)・失敗は「枠切れ」のみ）になる。
- **リージョン / アリーナ**: フェーズの終わりに一括解放する。

言葉だけだと味気ないので、その「枠の貸し借り」を最小限のコードで見ておこう。静的配列を起動時に空きリストへ繋ぎ、確保は pop、解放は push にする。ヒープには一切触れない。

```c
#define BLK 64
#define CAP 4
static uint8_t arena[BLK * CAP];   // 起動時に静的確保（ヒープを使わない）
static void *freelist;             // 空き枠の単方向リスト

static void *pool_alloc(void){     // O(1): pop。失敗は「枠切れ」のみ
  if (freelist == NULL) return NULL;
  void *blk = freelist;
  freelist = *(void**)blk;         // 枠の先頭に次ポインタを埋めてある
  return blk;
}
static void pool_free(void *blk){  // O(1): push
  *(void**)blk = freelist;
  freelist = blk;
}
```

枠を 4 つ確保し切ってから 5 個目を要求し、1 つ返してから再確保すると：

```
4 枠確保: a..d は非NULL=1, 5個目 e=(nil) (枠切れ)
b を返して再確保 f==b: 1
```

探索も併合も断片化もない。確保失敗は「枠が尽きた」一択で、起動時に `CAP` を決めた時点で最悪メモリ量も上界が確定する。これが安全クリティカルでメモリプールが好まれる理由だ。空きブロックの先頭に次ポインタを埋め込む手口は、[スラブ](buddy-slab.md)のフリーリストとまったく同じである。

TLSF のような「動的だが有界」なアロケータは、
これらが使えない（本当に実行時までサイズが読めない）場面の最後の手段、
という位置づけになる。

> [!NOTE]
> ガベージコレクションを持つ言語にも「リアルタイム GC」の研究があり、
> GC の停止時間を有界化する技術が積み上げられてきた。
> 本書の範囲外だが、動機は本章とまったく同じで、
> 「メモリ管理の最悪時間を有界にする」ことである。
> malloc 側の TLSF と GC 側のリアルタイム GC は、同じ問いへの別解だ。

## 組み込みでの実践的な勘どころ

最後に、組み込みでアロケータと付き合うときの実務的な注意を挙げておく。

第一に、確保失敗を必ず処理する。デスクトップでは
[オーバーコミット](os-interface.md)のせいで `malloc` が `NULL` を
返さないことも多いが、組み込みは物理メモリが小さく
オーバーコミットもないため、`NULL` は普通に返る。
返り値検査を省くと、本当に落ちる。

「デスクトップでも NULL を返させる」のは難しくない。`setrlimit(RLIMIT_AS, ...)` で仮想メモリ上限を 64 MiB に絞れば、組み込みのように `malloc` が素直に `NULL` を返すようになる。

```c
struct rlimit rl = { 64*1024*1024, 64*1024*1024 };  // 仮想メモリを 64 MiB に制限
setrlimit(RLIMIT_AS, &rl);
int n = 0; void *p;
while ((p = malloc(1*1024*1024)) != NULL) n++;       // 1 MiB ずつ確保
printf("%d 回成功し、%d 回目で NULL を返した\n", n, n+1);
```

```
61 回成功し、62 回目で NULL を返した
```

回数は環境により変わるが、確保が `NULL` で止まる事実は変わらない。物理メモリが小さくオーバーコミットもない組み込みでは、これが日常的に起きる。`malloc` の戻り値検査は省略してよい飾りではなく、必須の制御フローだ。

第二に、メモリプールをサイズ／用途ごとに分ける。
1 つの汎用ヒープに全部を混ぜると、ある用途の確保が
別の用途の断片化を引き起こし、問題の切り分けが難しくなる。
用途別プールなら「どのプールが枯渇したか」が一目で分かり、
最悪量も用途ごとに見積もれる。

第三に、最悪を測る。平均レイテンシではなく、
最悪レイテンシ（ヒストグラムの裾）と、長時間稼働後の
断片化（high-water の伸び）を測定する。
[断片化の章](fragmentation.md)で作った「high_water とピーク生存量の比」を
長時間流して、じわじわ伸びていないかを確認するのが定石だ。

## まとめ

リアルタイム・組み込みでは、平均ではなく**最悪**が要件になる。
TLSF[](#cite:masmano2004) は二段の segregated fit を
ビットマップで引くことで、確保と解放と併合すべてを真の $O(1)$ にし、
断片化の上界も保証する。一方、安全クリティカルの主流は
「実行中の動的確保を禁じ、静的確保やメモリプールで済ませる」流儀であり、
TLSF はそれが使えない場面の有界な動的解である。
共通する問いは「メモリ管理の最悪を有界にできるか」で、
これはリアルタイム GC とも地続きだ。
次章では、視点を現在から未来へ移し、
機械学習、ハードウェア、コンパクションといった最新研究を扱う。
