# サイズクラスと segregated fit

前章のフリーリストには、リストを「探索する」というコストが残っていた。
現代のアロケータは、ほとんどの要求でこの探索をしない。
答えは単純で、**サイズごとに専用のフリーリストを用意しておく**。
要求サイズから担当リストを計算し（数命令）、先頭を 1 つ外して返す（数命令）。
この方式を [segregated fit](#index:segregated fit)
（分離フィット、サイズ別フリーリスト）と呼び、
glibc の bin も、jemalloc・TCMalloc・mimalloc の中心機構も、すべてこれである。
この章では、その設計空間（クラスの刻み方、サイズからクラスへの変換、
クラス間でのメモリ融通）を扱う。

## サイズクラスという発想

[サイズクラス](#index:サイズクラス)（size class）とは、
「確保サイズをあらかじめ決めた離散的な系列に丸める」ことである。
たとえば系列が {16, 32, 48, 64, 80, ...} なら、`malloc(20)` は 32 バイトクラス、
`malloc(50)` は 64 バイトクラスとして扱う。

サイズクラスがもたらすものは速度だけではない。

- **探索の消滅**: クラスごとのリストは**同じサイズのブロックだけ**を含むので、
  「十分大きいブロックを探す」必要がなく、先頭を pop するだけでよい。
  push/pop だけなら単方向リストで足り、操作は $O(1)$ になる。
- **ぴったり再利用**: 同サイズなので分割の余りが出ない。
  実プログラムは同じサイズを大量に繰り返し確保する
  （構造体・オブジェクトのサイズは有限種類しかない）ため、的中率が高い。
- **ヘッダの削減**: 「このページはすべて 48 バイトブロック」と
  ページ単位で決めてしまえば、ブロックごとのサイズヘッダすら不要になる
  （ページ先頭のメタデータに 1 回書けばよい）。
  この形は特に segregated storage と呼ばれ、
  TCMalloc のスパン、jemalloc・mimalloc のスラブ/ページがこの設計である
  （[第 IV 部](part-libraries.md)で実物を見る）。

代償は[内部断片化](#index:内部断片化)（internal fragmentation）、
つまり丸めによる無駄である。20 バイトの要求に 32 バイトを渡せば
12 バイトが死蔵される。クラスの刻みを細かくすれば内部断片化は減るが、
クラス数が増え、リストごとに抱える在庫がかさむ。
**サイズクラス設計とは、この内部断片化とクラス数のトレードオフの調整**である。

## クラスの刻み方を設計する

最も単純な系列は 2 の冪 {16, 32, 64, 128, ...} だ。
クラス判定はビット演算 1 回だが、最悪ケースでは要求の直後の冪に丸めるため、
ほぼ **2 倍弱**（無駄率は最大で約 50%）を消費する。これは粗すぎる。

そこで実用アロケータは「各 2 倍区間を 2〜4 等分する」**等比＋等差**の系列を使う。
jemalloc と TCMalloc は 1 区間 4 分割（公比約 1.25）で、
最悪の無駄率は約 20% に抑えられる。Ruby で系列を生成してみよう。

```ruby
# 1 区間を 4 等分するサイズクラス系列（jemalloc 風の単純化版）
def size_classes(min_shift: 4, max: 4096)
  classes = (1..4).map { |i| (1 << min_shift) * i }   # 16,32,48,64 は等差
  shift = min_shift + 2
  while (base = 1 << shift) < max
    step = base / 4
    4.times { |i| classes << base + step * (i + 1) }  # base+25%, 50%, 75%, 100%
    shift += 1
  end
  classes.select { |c| c <= max }
end

p size_classes.first(16)
# => [16, 32, 48, 64, 80, 96, 112, 128, 160, 192, 224, 256, 320, 384, 448, 512]

# 各クラスの最悪の無駄率: (クラスサイズ - (1つ下のクラス+1)) / クラスサイズ
classes = size_classes
worst = classes.each_cons(2).map { |lo, hi| [(hi - lo - 1).fdiv(hi), hi] }.max
printf("最悪無駄率: %.1f%% (クラス %d)\n", worst[0] * 100, worst[1])
# => 最悪無駄率: 46.9% (クラス 32)

big = classes.each_cons(2).select { |_, hi| hi >= 128 }
             .map { |lo, hi| [(hi - lo - 1).fdiv(hi), hi] }.max
printf("128B 以上では: %.1f%% (クラス %d)\n", big[0] * 100, big[1])
# => 128B 以上では: 20.0% (クラス 2560)
```

等比に刻まれる 128 バイト以上では、たとえば 160 バイトクラスに 129 バイトの要求が
来ると 31/160 ≈ 19.4% が無駄になる。区間が大きいほどこの比率は 1/5 = 20% に漸近し
（コードが返す最悪はクラス 2560 の 511/2560 ≈ 20.0%）、おおむね「無駄は 2 割まで」に収まる。
一方、最小側の等差区間では 17 バイトの要求が 32 バイトクラスに丸まり、
比率でいえば 47% 近くが無駄になる。ただし絶対量は 15 バイトにすぎず、
これ以上刻みを細かくする（8 バイト刻みにする）かはアラインメント制約と
クラス数との相談になる。クラス数は 4 KiB までで 28 個。
この程度なら、クラスごとの在庫過剰も管理できる。

この「無駄率」は紙の上の話ではない。jemalloc には `nallocx(size, 0)` という、
「この要求は実際に何バイト確保されるか」を**確保せずに**問い合わせる関数がある。
これを使えば、本文の数値をそのまま実機で確かめられる。

```c
// gcc -O2 round.c /usr/lib/x86_64-linux-gnu/libjemalloc.so.2 -o round
#include <stdlib.h>
#include <stdio.h>
size_t nallocx(size_t size, int flags);
int main(void){
    size_t reqs[] = {17, 20, 50, 129, 200, 257};
    for (int i = 0; i < 6; i++){
        size_t got = nallocx(reqs[i], 0);
        printf("malloc(%3zu) -> 実サイズ %4zu  (無駄 %2zu, %.1f%%)\n",
               reqs[i], got, got - reqs[i], (got - reqs[i]) * 100.0 / got);
    }
    return 0;
}
```

```
malloc( 17) -> 実サイズ   32  (無駄 15, 46.9%)
malloc( 20) -> 実サイズ   32  (無駄 12, 37.5%)
malloc( 50) -> 実サイズ   64  (無駄 14, 21.9%)
malloc(129) -> 実サイズ  160  (無駄 31, 19.4%)
malloc(200) -> 実サイズ  224  (無駄 24, 10.7%)
malloc(257) -> 実サイズ  320  (無駄 63, 19.7%)
```

`malloc(17)` の 46.9%、`malloc(129)` の 31/160 ≈ 19.4% は、
本文で手計算した値と一桁まで一致する。最小側の等差区間ほど比率が大きく
（だが絶対量は 15 バイト）、128 バイト以上の等比区間では 2 割前後に収まる、
という全体像が実機でそのまま観察できる。

ここで生成した系列が机上の発明でないことは、手元の jemalloc に直接吐かせれば分かる。
`MALLOC_CONF=stats_print:true` を付けてプロセスを終了させると、bin（サイズクラス）の
一覧が表示される（既定では「実際に使われた bin」だけが出るので、下表は複数アリーナの
出力をまとめて空きクラスも補ったもの）。

```

```

ind 1 以降の `16, 32, 48, 64`（等差）、`80, 96, 112, 128`、`160, 192, 224, 256`、
`320, 384, 448, 512` …… は、本文の `size_classes` が返した 28 個の系列と完全に一致する。
唯一の違いは先頭の `8`（ind 0）で、jemalloc は本文の最小クラス 16 よりさらに下に
8 バイト bin を 1 つ持つ。`ind` がサイズ→クラス変換の行き先、すなわちこの章で言う
「クラス番号」である。本書の第 IV 部では、この表がどう引かれ、どう埋まるかを実物で追う。

サイズ→クラス番号の変換も設計ポイントだ。小さいサイズは
**8 バイト刻みの索引表**（`size >> 3` で表を引く）、大きいサイズは
最上位ビット位置（`clz` 命令）から計算する、というのが定番である。
変換は malloc の最頻実行パスなので、ここの数命令が全体性能に直結する。
malloc 専用ハードウェアを提案した Mallacc 論文[](#cite:kanev2017)が
加速対象に選んだ 3 操作のうち 1 つが、まさにこのサイズクラス変換だった
（[最新研究の章](research-frontiers.md)）。

> [!NOTE]
> 大サイズ側の変換が本当に「数命令」なのかは、コンパイル結果を見れば分かる。
> 最上位ビット位置（`__builtin_clzll`）から区間内 4 分割の番号を求める関数を
> `gcc -O2` でアセンブルすると、分岐もループもなく、命令列はこうなる。
>
> ```
> bsr  rax, rdi      ; 最上位ビット位置（clz の実体）
> xor  rax, 63
> ...
> sub  rdi, rdx      ; 区間先頭からのオフセット
> shr  rdi, cl       ; 4 分割のどこか
> ...
> ```
>
> `bsr`（bit-scan-reverse）1 命令に算術が十数個続くだけで、データ依存の分岐は
> 出ない。小サイズ側の `size >> 3` 表引きと合わせ、変換は数命令で済む。
> malloc の最頻パスにこれが何度も乗ることを思えば、Mallacc がここを
> ハードウェア加速の標的に選んだのも頷ける。

## segregated fit の原点としての quick fit

サイズ別リストの古典が Weinstock と Wulf の
**quick fit**[](#cite:weinstock1988) である。
頻出サイズ（例: 16〜128 バイト）にだけ「クイックリスト」を用意し、
ヒットすれば探索なしで返す。外れた要求は従来型の汎用アロケータ
（misc list）へ回す。さらに重要な性質として、クイックリスト上のブロックは
**併合しない**（遅延併合）。同じサイズがすぐまた要求される、
という観察に賭けているわけだ。

この「ホットパスは無併合の同サイズリスト、コールドパスは汎用機構」という
二段構えは、そのまま現代に生きている。glibc の fastbin と tcache は
quick fit のクイックリストの直系であり、
スレッドキャッシュ（[マルチスレッドの章](multithread.md)）も
「ヒットすれば数命令」という同じ哲学の産物である。

この「無併合の同サイズリスト」は glibc の tcache でそのまま観察できる。
同じサイズを 2 つ確保して解放し、もう一度確保すると、
探索もせず**直前に解放したブロックがそのまま**返ってくる（LIFO）。

```c
#include <stdlib.h>
#include <stdio.h>
int main(void){
    void *a = malloc(48), *b = malloc(48);
    printf("確保:   a=%p b=%p\n", a, b);
    free(a); free(b);              // a を push、続けて b を push（先頭は b）
    void *c = malloc(48), *d = malloc(48);  // 同じ 48B クラスから pop
    printf("再確保: c=%p d=%p\n", c, d);
    printf("c==b? %s   d==a? %s\n", c==b?"yes":"no", d==a?"yes":"no");
    return 0;
}
```

```
確保:   a=0x562e5b97e2a0 b=0x562e5b97e2e0
再確保: c=0x562e5b97e2e0 d=0x562e5b97e2a0
c==b? yes   d==a? yes
```

`free(a); free(b)` の後の `malloc` は `b`、次が `a` を返す。後入れ先出しで、
同サイズリストの先頭を pop しているだけだ（アドレスは環境により変わる）。
ブロックは併合されず、サイズ 48 のクラスに留まったまま即座に再利用される。
これが quick fit のクイックリストであり、glibc の fastbin・tcache の挙動である。

> [!NOTE]
> 用語を整理しておく。Wilson らのサーベイ[](#cite:wilson1995)では、
> サイズ別リストを使う方式の総称が **segregated free lists** で、
> その中に (1) 丸めたサイズのブロックを使い回す **segregated storage**
> （ページ単位でクラスを固定する simple segregated storage を含む）と、
> (2) リストはサイズ別だが、ブロックは分割・併合して融通する
> **segregated fit** がある。glibc の bin は (2)、
> TCMalloc/jemalloc/mimalloc の小サイズ機構は (1) に近い。
> 文献によって言葉の使い方が揺れるので、論文を読むときは定義を確認しよう。

## クラス間の在庫の融通

サイズ別リストの弱点は、**在庫がクラスごとに分断される**ことだ。
48 バイトのリストに 1 万個の空きがあっても、64 バイトの要求には使えない。
ヒープ全体では空きが余っているのに特定クラスだけ品切れする。
これは外部断片化の一種（あるいは「方針による断片化」）であり、
対策がアロケータの個性になる。

- **分割で補充**: 品切れクラスは、より大きい空き（または OS から取った
  新しいページ）を分割して補充する。glibc は unsorted bin・top チャンクから切り出す。
- **ページ単位の回収**: simple segregated storage 系では、
  「ページ内の全ブロックが空きになったらページをクラスから外し、
  別のクラスへ転用 or OS へ返す」。ページが空になるかどうかが勝負なので、
  **同じページに同じくらいの寿命のオブジェクトを集める**ことが
  断片化対策になる（この観察が後の研究[](#cite:maas2020)につながる。
  [最新研究の章](research-frontiers.md)参照）。
- **定期的なフラッシュ**: スレッドキャッシュやクイックリストの在庫を
  時々まとめて返し、併合の機会を作る（glibc の `malloc_consolidate`、
  TCMalloc のキャッシュ回収など）。

「クラスごとの在庫の持ちすぎ」をどう抑えるかは、
マルチスレッドのキャッシュ設計と絡んでさらに難しくなる。
Berger らはスレッドごとの無制限な在庫が引き起こすメモリ爆発を
**blowup** と名付けて定式化した[](#cite:berger2000)。
詳しくは[マルチスレッドの章](multithread.md)で扱う。

## 言語処理系の中のサイズクラス

サイズクラスは malloc ライブラリの専売特許ではない。
GC を持つ言語処理系のヒープにも同じ構造が現れる。

CRuby のオブジェクトヒープは、固定長スロットを敷き詰めた
ページ（heap page）の集合である。Ruby 3.2 から導入された
可変幅アロケーション（Variable Width Allocation）では、
スロット幅の異なる複数の **size pool**（40, 80, 160, 320, 640 バイト）が用意され、
オブジェクトは自分のサイズに合う pool のページに置かれる。
これは「ページ単位でクラスを固定する simple segregated storage」そのものだ。
それまで `String` や `Array` の中身は別途 `malloc` していたのが、
オブジェクト本体と同じスロットに**埋め込み**（embed）できるサイズの上限が広がり、
malloc 呼び出しとポインタ間接参照が減った。

```ruby
# CRuby で確認: オブジェクトが置かれているスロットのサイズ
require "objspace"
ObjectSpace.memsize_of("short")          # => 40   (40 バイト pool に収まる)
ObjectSpace.memsize_of("x" * 100)        # => 160  (160 バイト pool)
ObjectSpace.memsize_of("x" * 1000)       # => 1041 (埋め込み不可、malloc 行き)
```

`memsize_of` の値はビルドやバージョンで変わるが、
「小さい文字列は離散的なクラスサイズに丸まり、大きい文字列だけ
malloc 依存のバイト数になる」という形はどの環境でも観察できる。
処理系を書く人にとってサイズクラスは、malloc を**使う**ときにも
（丸めを意識した構造体設計）、malloc 的なものを**作る**ときにも
（オブジェクトヒープ設計）顔を出す基礎概念である。

## まとめ

サイズクラスは「探索を計算に置き換える」発想であり、
現代アロケータの速度はほぼこれで説明できる。
設計変数は (1) クラスの刻み（内部断片化の上限を決める）、
(2) サイズ→クラス変換の速さ、(3) クラス間の在庫融通（外部断片化を決める）の 3 つ。
そして「同じサイズを束ねてページ単位で管理する」発想を突き詰めると、
ブロックヘッダが消え、ページ自体が管理単位になる。
その究極形である**バディシステム**と**スラブアロケータ**を、次章で見よう。
