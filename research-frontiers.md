# 最新研究のフロンティア

最後の応用章では、ここまでの各章に収まりきらない新しい研究の流れを集める。
機械学習で確保を賢くする、ハードウェアで malloc を加速する、
C/C++ でも「移動」を実現して断片化を畳む、
そして 2025 年のプレプリントが示す「単純さへの回帰」。
動的メモリ割り当てが 60 年経っても生きた研究分野であることを、
具体的な論文で確かめよう。各テーマは独立に読める。

## 機械学習で寿命を予測する LLAMA

[断片化の章](fragmentation.md)で触れた古い経験則
（「寿命の近いオブジェクトを近くに置けば、まとめて死んでページが空く」）を、
機械学習で本気で実装したのが Maas らの
**LLAMA**（Learned Lifetime-Aware Memory Allocator）[](#cite:maas2020) である。
ASPLOS 2020 で発表された。

着想はこうだ。オブジェクトの寿命は、それを確保したコード位置
（コールスタック）とよく相関する。同じ場所で確保されるものは、
同じくらい生きて同じくらいで死ぬ傾向がある。
そこで LLAMA は、

- 大量の確保ログから、確保コンテキスト（スタック）から寿命クラスを
  予測するモデル（当初は LSTM、実行時には軽量化したもの）を学習する。
- 確保時に寿命クラスを予測し、**同じ寿命クラスのオブジェクトを
  同じヒュージページにまとめる**。
- 予測が外れた（思ったより長生きした）場合に備え、
  ヒュージページを徐々に「長寿命用」へ格上げする仕組みを持つ。

結果として、ヒュージページの断片化（[局所性の章](locality.md)で見た
「住人 1 人で立ち退けない」問題）を大きく減らせた。
これは「アロケータは将来を知らない」という[最初の章](introduction.md)で
述べた大前提に、学習で部分的に挑む試みである。
モデルの推論コストを確保のホットパスにどう収めるかが鍵で、
論文はそのための工夫（予測の量子化、サンプリング）も論じている。
寿命予測という発想自体は Barrett と Zorn[](#cite:barrett1993) が
1993 年に提案済みで、30 年越しに機械学習で本格化したと見ることもできる。

なぜヒュージページの断片化がそれほど効くのか、数で見ておこう。本書の環境のヒュージページは 2 MiB（`/sys/kernel/mm/transparent_hugepage/hpage_pmd_size` が `2097152`）で、ここに小さなオブジェクトを敷き詰めると次のようになる。

| オブジェクトサイズ | 2 MiB ヒュージページに入る個数 |
|---:|---:|
| 16 B | 131072 個 |
| 64 B | 32768 個 |
| 256 B | 8192 個 |
| 4096 B | 512 個 |

64 B なら 1 枚に 32768 個。このうち **たった 1 個でも生き残れば、ページ全体（残り 32767 スロット分の物理メモリ）を OS へ返せない**。LLAMA が「同じ寿命クラスを同じヒュージページへ」と寄せるのは、この 1/32768 の生存確率を実質ゼロに近づけて、ページ単位でまとめて空けるためである。

> [!NOTE]
> 「学習したモデルをアロケータに積む」アプローチには、再現性や
> ワークロード変化への頑健性という課題もある。
> それでも、確保パターンというプログラム実行の統計的構造を
> 明示的に使う方向は、今後のアロケータ研究の太い流れになりそうだ。

## C/C++ でオブジェクトを動かす Mesh のコンパクション

[断片化の章](fragmentation.md)で、「C/C++ は生ポインタがあるので
オブジェクトを移動できず、$\log n$ 倍の最悪断片化を免れない」と述べた。
**Mesh**[](#cite:powers2019)（PLDI 2019）は、この常識を
仮想メモリのトリックで破った、鮮やかな研究である。

ポインタを書き換えずにオブジェクトを「動かす」にはどうするか。
Mesh の答えは、「**仮想アドレスはそのまま、物理ページだけ統合する**」だ。

- 同じサイズクラスの 2 つのページがあり、使用中スロットの位置が
  互いに重ならないとき、両者を 1 つの物理ページに重ね合わせできる。
- 2 つの仮想ページを同じ物理ページにマップし直せば、
  ユーザのポインタ（仮想アドレス）は一切変わらないまま、
  物理メモリは半分になる。
- どのページ対が重ね合わせ可能かを、ランダム化と組み合わせて効率的に探す
  （任意のマッチングは難しいが、確率的に十分見つかることを理論的に保証している）。

これにより、移動なしでは不可能なはずの物理メモリのコンパクションが、
C/C++ のままで実現する。Firefox などで実際にメモリ削減を確認している。
「ポインタは動かせないが、ポインタの指す先の物理実体は動かせる」という
仮想メモリの抽象を突いた、malloc 研究の白眉のひとつだ。

Mesh の核心（「仮想アドレスは固定したまま物理ページだけ統合する」）は、`memfd` を 2 か所へマップするだけで体感できる。同じサイズクラスの 2 ページ A と B の使用スロットが重ならない（A はスロット 0、B はスロット 1 だけ生存）状況を作り、両者を 1 枚の物理ページへ重ね合わせてみる。

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <sys/mman.h>
#include <unistd.h>
int main(void) {
    long ps = sysconf(_SC_PAGESIZE);
    int fd = memfd_create("phys", 0);   /* 物理実体は 1 ページだけ */
    if (ftruncate(fd, ps) != 0) return 1;
    /* 仮想ページ A・B を別々の仮想アドレスへ、同一物理へマップ */
    char *A = mmap(NULL, ps, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
    char *B = mmap(NULL, ps, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
    printf("A=%p  B=%p  (別ページ)\n", (void*)A, (void*)B);
    A[0]    = 'X';   /* A 側の生存スロット */
    B[2048] = 'Y';   /* B 側の生存スロット(A と重ならない) */
    printf("B[0]='%c'  A[2048]='%c'  物理 2->1、仮想は不変\n", B[0], A[2048]);
}
```

```
A=0x7248ec21d000  B=0x7248ec21c000  (別ページ)
B[0]='X'  A[2048]='Y'  物理 2->1、仮想は不変
```

A への書き込み `'X'` が B から、B への `'Y'` が A から透けて見える。確かに **同一の物理ページ** になっている。それでいて A と B の仮想アドレスは別のままで、ユーザのポインタは一切書き換えていない。実際の Mesh はこれをサイズクラス単位のページ集合に対して、重ね合わせ可能なペアをランダム化探索で見つけて行う。アドレスは環境により変わる。

## ハードウェアで malloc を速くする Mallacc

malloc は呼ばれる回数が桁違いに多いので、
1 回あたり数命令の削減でも全体では大きい。
そこに目をつけ、**CPU に malloc 専用命令を足す**ことを提案したのが
Kanev らの **Mallacc**[](#cite:kanev2017)（ASPLOS 2017）である。

彼らはまず、データセンターのアプリで malloc がどれだけ
CPU 時間を食っているか（無視できない割合だった）を測り、
ホットパスの中身を分解した。すると時間の大半が、

- サイズクラスの計算（[サイズクラスの章](segregated-lists.md)）、
- スレッドキャッシュのフリーリスト先頭の取り出し、
- そのためのロードと分岐、

という少数の操作に集中していた。Mallacc は、これらを
小さな専用ハードウェア（確保のためのキャッシュ的な構造と数個の命令）で
肩代わりさせ、シミュレーションで malloc のホットパスを大きく短縮した。
「アロケータのソフトウェア設計はほぼ枯れた。次はハードウェア」という
問題意識を示した点で重要な研究だ。CHERI（[セキュリティの章](security.md)）も
ハードウェア側からメモリ管理に介入する流れで、
malloc とハードウェアの境界が近年あらためて動いている。

## 2025 年の地平に見る単純さへの回帰と専用化

最後に、ごく最近のプレプリントを 2 本紹介する。
どちらも 2025 年のもので、今まさに動いている研究の空気を伝える。

**ExGen-Malloc**[](#cite:li2025exgen) は、
「マルチスレッド対応で複雑化した現代アロケータは、
シングルスレッドのアプリではかえって重い」という問題提起から出発する。
スレッドキャッシュや同期のための機構は、1 スレッドしかないなら純粋な
オーバーヘッドだ。そこで単一スレッド向けに割り切った設計を取り、
SPEC CPU2017 で dlmalloc 比 1.17 倍の高速化、また mimalloc に対しては約 6% のメモリ削減を報告している。
[マルチスレッドの章](multithread.md)で見た「並列対応の代償」を
逆から照らす、面白い視点である。

「1 スレッドしかないなら同期は純粋なオーバーヘッド」を、jemalloc で数字にしてみよう。単一スレッドで 64 B の `malloc`/`free` を 5000 万回まわし、スレッドキャッシュを `MALLOC_CONF` で入り切りする。`tcache:false` にすると、毎回の確保がアリーナのロックを通る（＝マルチスレッド対応の機構をそのまま踏む）。

```console
# tcache_demo.c: 確保したポインタを volatile で受けて足し込む
# （単に malloc/free するだけだと -O2 でループごと消えてしまう）
$ gcc -O2 tcache_demo.c -o tcache_demo
$ LD_PRELOAD=/usr/lib/x86_64-linux-gnu/libjemalloc.so.2 \
    MALLOC_CONF="tcache:true"  ./tcache_demo   # 約 0.3 秒
$ LD_PRELOAD=/usr/lib/x86_64-linux-gnu/libjemalloc.so.2 \
    MALLOC_CONF="tcache:false" ./tcache_demo   # 約 9 秒
```

同じ単一スレッドのワークロードで **約 25 倍** の差がつく。`tcache` という前段のキャッシュが、奥にあるロックを毎回触らずに済ませているだけで、機構そのものは存在し続けている。ExGen-Malloc が「単一スレッド向けに割り切れば、この前段すら要らない」と言うのは、まさにこの隠れたコストを設計段階から消すという主張だ。時間は環境により変わる。

**SpeedMalloc**[](#cite:li2025speed) は逆に、
マルチスレッドをハードウェアで攻める。
専用の軽量コアにメモリ確保の仕事を肩代わりさせ、
アロケータのメタデータをユーザデータから物理的に切り離すことで、
キャッシュ汚染と同期コストを下げる。複数の既存アロケータに対して
1.15〜1.75 倍の高速化を報告している。Mallacc の「ハードウェア支援」の
流れを、専用コアという形で受け継いだものと言える。

この 2 本が示すのは、動的メモリ割り当てが
「単純化（専用化）」と「ハードウェア化」という両方向に
いまも広がり続けているということだ。
1965 年のバディシステム[](#cite:knowlton1965)から数えて 60 年、
malloc はまだ枯れていない。

> 最新研究を追うなら、ISMM（International Symposium on Memory Management）、
