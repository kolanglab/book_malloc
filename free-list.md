# フリーリストと配置方針 — first-fit、best-fit、境界タグ

`free` で返ってきたメモリを、アロケータはどう記録し、
次の `malloc` でどう選んで再利用するのか。
この章では、その最も基本的な答えである[フリーリスト](#index:フリーリスト)
（free list）と、選び方の方針（first-fit / best-fit / next-fit）、
そして隣り合う空きをまとめる**境界タグ法**を学ぶ。
Ruby で小さなシミュレータを書き、方針の違いが断片化にどう効くかを実験する。

## ポリシーとメカニズムを分けて考える

Wilson らの古典サーベイ[](#cite:wilson1995)は、アロケータの議論を
3 つの層に分けることを勧めている。本書でもこの語彙を使う。

- **戦略（strategy）**: どんな性質を狙うか。「将来の要求に備えて大きな連続空きを温存する」など。
- **ポリシー（policy）**: 具体的な選択規則。「条件を満たす最も低いアドレスの空きを選ぶ」など。
- **メカニズム（mechanism）**: ポリシーを実装するデータ構造とアルゴリズム。
  「単方向連結リストを先頭から線形探索」など。

たとえば「best-fit は断片化が少ない」は**ポリシー**の性質であり、
それを線形探索で実装するか平衡木で実装するか（＝メカニズム）は別の問題だ。
論文を読むときも実装を読むときも、この区別があると頭が整理される。

## 暗黙のリストから明示的フリーリストへ

最も素朴なヒープは、メモリ全体を**ヘッダつきブロックの列**として管理する。
各ブロックの先頭に「サイズ」と「使用中か空きか」を記録しておけば、
先頭から順にヘッダをたどって全ブロックを列挙できる。
これを**暗黙のフリーリスト**（implicit free list）と呼ぶ。
空きブロックを探すのに使用中ブロックも全部なめる必要があるため、
ブロック総数 $n$ に対して探索は $O(n)$ となり、実用には遅すぎる。

そこで、**空きブロックの中に**「次の空きブロックへのポインタ」を埋め込み、
空きだけを連結する。これが**明示的フリーリスト**（explicit free list）である。

```
使用中ブロック:                 空きブロック:
+--------+----------------+    +--------+------+------+--------+
| header | ユーザデータ     |    | header | next | prev | (未使用) |
+--------+----------------+    +--------+------+------+--------+
```

ポインタを置く場所は空きブロックの**本体**（どうせ誰も使っていない）なので、
追加のメモリは要らない。ただしこの設計から、
「空きブロックには next/prev とヘッダを格納できるだけの最小サイズが要る」
という制約が生まれる。64 ビット環境で `malloc(1)` にも 32 バイト級の
チャンクが使われる（[前章で実測した](inside-malloc.md)）のは、これが理由だ。

## 配置方針 — どの空きを使うか

空きが複数あるとき、どれを選ぶか。古典的な選択肢は 3 つある
[](#cite:knuth1997)[](#cite:wilson1995)。

- **first-fit**: リストを先頭からたどり、**最初に見つかった**十分大きい空きを使う。
- **next-fit**: first-fit の変種。前回見つけた位置から探索を再開する
  （リストを一周する円環とみなす）。探索の偏りを避け、平均探索長を縮める狙い。
- **best-fit**: **余りが最小になる**空きを使う。素朴には全リスト走査が必要。

直感的には「best-fit は几帳面で無駄が少ないが遅い、first-fit は雑だが速い」
と見えるが、研究の結論はもう少し面白い。

言葉だけでは分かりにくいので、両者が違うブロックを選ぶ最小例を見ておく。空きが `[[0, 40], [50, 20]]` の 2 つ（間は使用中ブロックで隔てられ、併合できない）の状態で `malloc(20)` を要求すると、こうなる。

```
first_fit: malloc(20) => addr 0   後の free_list=[[20, 20], [50, 20]]
best_fit:  malloc(20) => addr 50  後の free_list=[[0, 40]]
```

first-fit は先頭の `[0, 40]` を切り、20 の端切れと 20 の穴の **2 つ**を残す。best-fit はぴったりの `[50, 20]` を丸ごと使い切り、大きい `[0, 40]` を **温存**する。「将来の大きな要求に備えて連続空きを残す」という best-fit の戦略が、たった 1 回の確保にも現れている。これが「best-fit は無駄が少ない」の中身だ。

1970 年代の解析・実験研究[](#cite:shore1975)[](#cite:bays1977)から
すでに、**アドレス順（address-ordered）の first-fit は best-fit と同等に良い**
ことが知られていた。Wilson らの再評価[](#cite:wilson1995)と
Johnstone らの実プログラム実験[](#cite:johnstone1998)はこれを強く支持し、
さらに **next-fit は明確に劣る**（解放直後の領域を転々と使い、
さまざまな寿命のオブジェクトをヒープ中に撒き散らすため）ことも示した。
「探索開始点を動かして速くする」という一見無害な変更が
断片化を悪化させる——ポリシーの違いは侮れない。

なお最悪ケースの理論では、best-fit は first-fit より**悪い**ことが
証明されている（Robson の結果[](#cite:robson1977)。
[断片化の章](fragmentation.md)で数式ごと扱う）。
平均（実プログラム）と最悪（敵対的入力）で順位が入れ替わるのも、
この分野の味わい深いところだ。

## 分割と併合 — 空きの加工

選んだ空きが要求より大きければ、前半を切って渡し、
後半を新しい空きとしてリストへ戻す。これが**分割**（splitting）である。
分割を無制限にやると小さな空きが無数にできるので、
「余りが最小ブロックサイズ未満なら分割せず丸ごと渡す」のが普通だ。

逆方向の操作が[併合](#index:併合)（coalescing）である。
`free` されたブロックの**物理的に隣**のブロックも空きなら、
くっつけて 1 つの大きな空きにする。併合を怠ると、
一度小さく切られたメモリは小さいままで、大きな要求に応えられなくなる
——外部断片化の典型的な発生メカニズムである。

問題は「前の隣」をどう見つけるかだ。ヘッダはブロックの先頭にあるので、
**次の**ブロックへは「自分のアドレス＋サイズ」で飛べるが、
**前の**ブロックの先頭は分からない。
Knuth が広めた解法が[境界タグ](#index:境界タグ)（boundary tag）法である
[](#cite:knuth1997)：各ブロックの**末尾にもサイズを複製**しておく（フッタ）。
`free(p)` のとき、直前の 1 ワードを読めば前のブロックのサイズが分かり、
その先頭へ $O(1)$ で戻れる。

```
低位アドレス →
... | size | (block A)        | size | size | (block B) ... 
      ^Aのヘッダ                ^Aのフッタ ^Bのヘッダ
```

フッタは 1 ブロックあたり 1 ワードのオーバーヘッドだが、
「**使用中**ブロックにはフッタ不要（前のブロックが空きのときだけ読まれる。
前が使用中なら戻る必要がない）」という最適化が効く。
ヘッダのサイズ値は 16 の倍数なので下位ビットが空いており、
そこに「前のブロックは使用中か」フラグ（glibc の `PREV_INUSE`）を格納すれば、
使用中ブロックはフッタ分までユーザデータに使える。
glibc malloc のチャンクがまさにこの設計である（[glibc malloc の章](glibc-malloc.md)）。

この設計は実物で覗ける。glibc では返ってきたポインタの 1 ワード手前に「チャンクサイズ ＋ 下位フラグ」が入っている。下位 3 ビットを落とせばサイズ、最下位ビットが `PREV_INUSE` だ。

```c
size_t *h = (size_t *)p - 1;          /* user ポインタの直前 */
size_t v = *h;
printf("chunksize=%zu  PREV_INUSE=%zu\n", v & ~(size_t)7, v & 1);
```

```
malloc(1)    size_word=0x21  -> chunksize=32    PREV_INUSE=1
malloc(24)   size_word=0x21  -> chunksize=32    PREV_INUSE=1
malloc(1000) size_word=0x3f1 -> chunksize=1008  PREV_INUSE=1
```

`0x21` は $32 \,|\, 1$、すなわち「サイズ 32・直前は使用中」を 1 ワードに詰めたものだ（glibc 2.39, x86-64）。`malloc(1)` も `malloc(24)` も同じ 32 バイトチャンクに丸められている——前節の「next/prev とヘッダを置ける最小サイズ」の制約が、ここに現れている。

併合を `free` のたびに行うのが**即時併合**（immediate coalescing）、
後回しにするのが**遅延併合**（deferred coalescing）である。
同じサイズの確保と解放が繰り返される場合、即時併合は
「くっつけてはまた切る」を繰り返す無駄になる。
glibc の fastbin や各種のスレッドキャッシュは、意図的に併合を遅らせる
遅延併合の実例だ。代償として、在庫が偏ると断片化が進む。

## Ruby でフリーリストアロケータを書く

ここまでの部品——明示的フリーリスト、first/best-fit、分割、アドレス順管理——を
Ruby で組み立てる。実メモリの代わりに「アドレス＝整数、ヒープ＝区間の集合」と
抽象化したシミュレータである。

```ruby
# 空きブロック: [先頭アドレス, サイズ] をアドレス順で保持する
class FreeListAllocator
  attr_reader :free_list

  def initialize(heap_size, policy: :first_fit)
    @heap_size = heap_size
    @policy    = policy
    @free_list = [[0, heap_size]]   # 最初はヒープ全体が 1 つの空き
    @allocated = {}                 # 使用中: アドレス => サイズ
  end

  def malloc(size)
    idx = case @policy
          when :first_fit then @free_list.index { |_, sz| sz >= size }
          when :best_fit  then
            cands = @free_list.each_index.select { |i| @free_list[i][1] >= size }
            cands.min_by { |i| @free_list[i][1] }
          end
    return nil unless idx                  # 十分な空きがない
    addr, sz = @free_list[idx]
    if sz == size then @free_list.delete_at(idx)       # ぴったり
    else @free_list[idx] = [addr + size, sz - size]    # 分割して残りを戻す
    end
    @allocated[addr] = size
    addr
  end

  def free(addr)
    size = @allocated.delete(addr) or raise "double free at #{addr}"
    i = @free_list.index { |a, _| a > addr } || @free_list.size
    @free_list.insert(i, [addr, size])     # アドレス順を保って挿入
    coalesce(i)
  end

  private

  def coalesce(i)   # 挿入位置 i の前後と併合を試みる（境界タグ相当の処理）
    if i + 1 < @free_list.size &&
       @free_list[i][0] + @free_list[i][1] == @free_list[i + 1][0]
      @free_list[i][1] += @free_list[i + 1][1]
      @free_list.delete_at(i + 1)
    end
    if i > 0 && @free_list[i - 1][0] + @free_list[i - 1][1] == @free_list[i][0]
      @free_list[i - 1][1] += @free_list[i][1]
      @free_list.delete_at(i)
    end
  end
end
```

配列の挿入・削除を使っているので効率は本物と違うが、
**ポリシーの挙動**は忠実である。動かしてみよう。

```ruby
a = FreeListAllocator.new(100)
x = a.malloc(30)   # => 0
y = a.malloc(30)   # => 30
z = a.malloc(30)   # => 60
a.free(y)          # 真ん中に 30 の穴
a.free(x)          # x と y の空きが併合される
a.free_list        # => [[0, 60], [90, 10]]
a.malloc(50)       # => 0   併合のおかげで 50 が入る！
```

もし `coalesce` を呼ばないようにすると、`free_list` は
`[[0, 30], [30, 30], [90, 10]]` のままで、`malloc(50)` は
合計 70 の空きがあるのに `nil` を返す。
**合計は足りているのに連続していないから失敗する**——
外部断片化の最小再現例である。

## 実験 — first-fit と best-fit を戦わせる

ランダムな確保・解放の負荷でポリシーを比べてみる。

```ruby
def stress(policy, seed: 42, ops: 2000)
  rng  = Random.new(seed)
  a    = FreeListAllocator.new(100_000, policy: policy)
  live = []
  fail_count = 0
  ops.times do
    if live.empty? || rng.rand < 0.6
      addr = a.malloc(rng.rand(16..512))
      addr ? live << addr : fail_count += 1
    else
      a.free(live.delete_at(rng.rand(live.size)))
    end
  end
  frags = a.free_list.size
  largest = a.free_list.map { |_, s| s }.max
  printf("%-10s 失敗 %3d / 空き断片 %3d / 最大空き %6d\n",
         policy, fail_count, frags, largest)
end

stress(:first_fit)
stress(:best_fit)
```

筆者の環境での出力例（seed 固定なので Ruby が同じなら再現する）:

```
first_fit  失敗  70 / 空き断片 152 / 最大空き    172
best_fit   失敗  56 / 空き断片 144 / 最大空き    399
```

この負荷では best-fit がやや優勢だが、大差はない。
ここで「ランダム負荷の結果を信じすぎるな」というのが
Wilson らの警告[](#cite:wilson1995)だったことを思い出してほしい。
実プログラムの確保パターンは、ランダムどころか
**フェーズ性**（一斉に確保して一斉に捨てる）と
**サイズの偏り**（ごく少数のサイズが大半を占める）に支配されている。
シードや確保サイズ分布、解放の偏り（古いものから解放する等）を変えて、
順位が入れ替わる負荷を探してみてほしい。それがそのまま
1990 年代の研究の追体験になる。

ところで、本章は「next-fit は明確に劣る」と書いたのに、この実験には入れていない。`malloc` に next-fit（前回見つけた位置から探索を再開する）を足し、シードの当たり外れをならすため seed を 1..20 で振って平均をとると、傾向がはっきりする。

```
first_fit  平均 失敗  44.4 / 断片 150.6 / 最大空き 1039.3
best_fit   平均 失敗  36.9 / 断片 136.4 / 最大空き 1300.2
next_fit   平均 失敗  51.0 / 断片 152.1 / 最大空き  494.4
```

断片数は first-fit とほぼ同じ・失敗はやや多い程度なのに、**最大空きだけが際立って小さい**（1039 → 494 と半分以下）。next-fit は解放直後の領域を転々と使うため、寿命の違うオブジェクトをヒープ中に撒き散らし、大きな連続空きを保てない。「探索開始点を動かす」という速度目的の小細工が連続性を壊す、という本文の主張が数字で見える。

## 探索を速くするメカニズム

ポリシーを固定しても、メカニズムには工夫の余地がある。
best-fit の線形探索は遅いので、空きブロックを**サイズをキーにした
データ構造**で持てばよい。歴史的には Cartesian 木によるアロケータ
[](#cite:stephenson1983)が早い例で、dlmalloc は大きなチャンク用に
サイズの 2 分トライ（treebin）を使い、TLSF はサイズをビット演算で
2 段の表に引く（[リアルタイムの章](realtime.md)）。
しかし現代の主流は、もっと割り切った答え——
「**そもそもよく出るサイズごとに専用リストを持てば、探索自体が消える**」——であり、
それが次章の[サイズクラスと segregated fit](segregated-lists.md) である。

> [!NOTE]
> この章のシミュレータは、ポリシー実験用に単純化している。
> 実物との主な違いは (1) ヘッダ・アラインメントを無視している、
> (2) 配列でなくリンクリスト＋境界タグで $O(1)$ 挿入削除する、
> (3) 実物はサイズ別リスト（次章）と併用する、の 3 点である。
> 「実物の縮図」としては [dlmalloc の章](dlmalloc.md)が最良の教材になる。
