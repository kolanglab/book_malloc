# 断片化 — 測り方、理論、そして「解決済み？」

アロケータの最大の敵は[断片化](#index:断片化)（fragmentation）である。
「メモリは余っているのに要求に応えられない」「`free` したのに
プロセスのメモリが減らない」という現象は、すべて断片化に帰着する。
この章では、断片化を**定義し、測り、理論の限界を知り、
実プログラムでの実態を確かめる**。
断片化は 60 年にわたって研究され、最悪ケースの厳しい下界が証明される一方、
「実用上は解決済みでは」という主張も生まれた、奥行きのあるテーマである。

## 内部断片化と外部断片化

断片化には 2 種類ある。すでに前の章で両方に触れたが、ここで厳密に分ける。

**内部断片化**（internal fragmentation）は、
**渡したブロックの中**の無駄である。サイズクラスへの丸め
（[サイズクラスの章](segregated-lists.md)）、アラインメントのための余白、
チャンクヘッダなど、「要求より大きく確保した分」がこれにあたる。
ブロック単位で閉じているので、測るのも見積もるのも比較的やさしい。

実際に glibc（2.39）で測ると、内部断片化は数字としてはっきり見える。`malloc_usable_size` は、要求サイズに対してアロケータが**本当に**割り当てた使用可能バイト数を返す。

```c
#include <stdio.h>
#include <stdlib.h>
#include <malloc.h>
int main(void) {
    size_t reqs[] = {1, 8, 24, 25, 33, 100, 128, 200, 500, 1000, 4096};
    for (size_t i = 0; i < sizeof reqs / sizeof *reqs; i++) {
        size_t r = reqs[i], u = malloc_usable_size(malloc(r));
        printf("%5zu %5zu %5zu (%4.1f%%)\n", r, u, u - r, (double)(u - r) / u * 100);
    }
}
```

```text
 要求  実際   無駄
    1    24    23 (95.8%)
    8    24    16 (66.7%)
   24    24     0 ( 0.0%)
   25    40    15 (37.5%)
   33    40     7 (17.5%)
  100   104     4 ( 3.8%)
  128   136     8 ( 5.9%)
  200   200     0 ( 0.0%)
  500   504     4 ( 0.8%)
 1000  1000     0 ( 0.0%)
 4096  4104     8 ( 0.2%)
```

最小チャンクが 24 バイト、刻みが 16 バイトなので、`malloc(1)` は 23 バイトも無駄になる一方、要求が大きくなれば無駄の比率はすぐ下がる。チャンクヘッダの 8 バイトぶん（`malloc(128)` が 136 になる類）も内部断片化の一部だ。小さく細かい確保ほど内部断片化が効く、という経験則がそのまま数字に出ている。

**外部断片化**（external fragmentation）は、
**ブロックとブロックの間**の無駄である。空きの総量は足りているのに、
それらが小さく散らばっていて連続要求に応えられない状態
（[フリーリストの章](free-list.md)の `malloc(50)` が失敗する例）を指す。
こちらは「いつ・どの順で確保解放されたか」という履歴に依存し、
ずっと扱いが難しい。

両者はトレードオフの関係にある。内部断片化を減らそうとクラスを細かく刻むと、
クラスごとの在庫が増えて外部断片化（クラス間で融通できない空き）が増える。
逆も然り。アロケータ設計とは、この 2 つの綱引きの落としどころ探しでもある。

## 断片化をどう測るか

「断片化率 30%」と言うとき、何を分子・分母にしているのか。
実は定義が複数あり、Wilson らのサーベイ[](#cite:wilson1995)は
この曖昧さ自体を問題視している。代表的な測り方は次のとおりだ。

ある時点で、アプリが使っている量を $L$（live bytes）、
アロケータが OS から取っている量を $H$（heap bytes）とすると、

```math
F = \frac{H - L}{L} \quad\text{または}\quad F' = 1 - \frac{L}{H}
```

がよく使われる。だが「いつの $L$ と $H$ か」で値が変わる。
ピーク時の $L$ を使うのか、各時点の最大 $H$ を使うのか、
平均を取るのか。Wilson らは「**生存量のピークに対する、ヒープ高水位の比**」
を推奨している。この曖昧さがあるため、論文の断片化数値を比べるときは
必ず定義を確認しなければならない。

Ruby で、第 II 部のフリーリストシミュレータに測定を足して観察してみよう。

```ruby
# free-list の章の FreeListAllocator を流用し、ピーク live と high-water を測る
def fragmentation_trace(policy, seed: 7, ops: 5000, heap: 1_000_000)
  rng  = Random.new(seed)
  a    = FreeListAllocator.new(heap, policy: policy)
  live_bytes = 0; peak_live = 0
  sizes = {}            # addr => size
  high_water = 0        # 使用された最大アドレス（簡易な heap high-water）
  ops.times do
    if sizes.empty? || rng.rand < 0.55
      sz = rng.rand(8..2048)
      if (addr = a.malloc(sz))
        sizes[addr] = sz; live_bytes += sz
        high_water = [high_water, addr + sz].max
        peak_live  = [peak_live, live_bytes].max
      end
    else
      addr = sizes.keys.sample(random: rng)
      live_bytes -= sizes.delete(addr); a.free(addr)
    end
  end
  printf("%-10s peak_live=%d  high_water=%d  F=%.1f%%\n",
         policy, peak_live, high_water, (high_water - peak_live).fdiv(peak_live) * 100)
end

fragmentation_trace(:first_fit)
fragmentation_trace(:best_fit)
```

この測定では「high_water（使った最も高いアドレス）がピーク生存量を
どれだけ上回るか」を断片化とみなしている。実際に走らせると、
ランダム負荷でも数〜十数 % の差が出る。負荷の偏らせ方
（サイズ分布、解放順）で結果が大きく動くことを、ぜひ自分の手で確かめてほしい。

シミュレータだけでなく、本物の glibc malloc でも同じことが起きる。256 バイトのブロックを 10 万個確保し、**1 つおきに**解放してみよう。空き総量は半分になるが、空きが隣同士でつながらないため、OS から取った量（system bytes）は 1 バイトも減らない。

```c
#include <stdio.h>
#include <stdlib.h>
#include <malloc.h>
int main(void) {
    enum { N = 100000, SZ = 256 };
    char **p = malloc(N * sizeof *p);
    for (int i = 0; i < N; i++) p[i] = malloc(SZ);
    fprintf(stderr, "=== 全生存 (live ≈ %.1f MiB) ===\n", (double)N*SZ/(1<<20));
    malloc_stats();
    for (int i = 0; i < N; i += 2) free(p[i]);   /* 1 つおきに解放 */
    fprintf(stderr, "\n=== 1 つおきに解放 (live は半分) ===\n");
    malloc_stats();
}
```

```text
=== 全生存 (live ≈ 24.4 MiB) ===
Arena 0:
system bytes     =   27303936
in use bytes     =   27200656
...
=== 1 つおきに解放 (live は半分) ===
Arena 0:
system bytes     =   27303936   ← 変わらない
in use bytes     =   13602560   ← 半分になった
```

`in use bytes` は半分に落ちたのに `system bytes` は `27303936` のまま動かない。これが「`free` したのにプロセスのメモリが減らない」（章冒頭）の正体だ。解放したブロックの両隣がまだ生きているので併合が起きず、空きはブロックサイズの穴のまま散らばっている。後の Robson の下界で出てくる「市松模様に解放する罠」を、実機で再現したものでもある。

## 理論の限界 — Robson の下界

断片化は、賢いアロケータを作ればゼロにできるのだろうか。
答えは**否**で、これは 1970 年代に厳密に証明されている。

最大ブロックサイズを $n$、生存データのピーク量を $M$ とする。
このとき、**どんなアロケータでも**、最悪の確保解放列に対して
$M \cdot \log_2 n$ のオーダーのメモリを要求しうる
[](#cite:robson1971)。直感的には、敵対者は
「いろいろなサイズを確保し、市松模様に 1 つおきに解放する」ことで、
空きを連続させない罠を仕掛けられる。
この下界は、コンパクション（移動）をしないアロケータには本質的なものだ。

さらに Robson は、具体的な方針の最悪ケースも解析した
[](#cite:robson1977)。それによると **best-fit の最悪ケースは
first-fit より悪い**。直感に反するこの結果は、
「平均では best-fit が優秀」という[フリーリストの章](free-list.md)の
実験と矛盾しない。**平均と最悪では順位が入れ替わりうる**——
これが断片化理論の最も重要な教訓である。

> [!NOTE]
> $\log n$ 倍という下界は「コンパクションをしない」前提で効く。
> オブジェクトを移動してよい GC（コピー GC など）はこの限界を超えられる。
> C/C++ は生ポインタがあるため一般には移動できないが、
> 仮想メモリのトリックで「物理的に併合する」Mesh[](#cite:powers2019) という
> 研究もある（[最新研究の章](research-frontiers.md)）。
> malloc が断片化と格闘し続けるのは、この移動の禁が効いているからだ。

## 実プログラムでの現実 — 「断片化問題は解決済み？」

理論は厳しい。では実際のプログラムでは断片化はどれくらい起きるのか。
ここで Johnstone と Wilson の 1998 年の論文
「The Memory Fragmentation Problem: Solved?」[](#cite:johnstone1998)が
衝撃を与えた。彼らは実在の C/C++ プログラムを 8 本選び、
複数のアロケータの断片化を**実トレースで**測定した。結論はこうだ。

- アドレス順 first-fit と best-fit という「良い方針」を、
  即時併合・分割と組み合わせて使えば、**実プログラムの断片化は
  ならして 1% 程度**しかない。
- 過去の研究で断片化が大きく見えていたのは、多くが
  **合成ランダム負荷**で評価していたためである。実プログラムは
  サイズが少数種に偏り、寿命にも規則性があるので、ずっと素直に振る舞う。
- 悪名高い next-fit や、2 の冪へ丸める方式（バディや単純な
  segregated storage）は、実プログラムでも明確に断片化が大きい。

タイトルの問い「解決済みか？」への彼ら自身の答えは、
「良い方針を選べば、実用上はほぼ解決済み」だった。
ただしこれは「(a) 単一スレッド、(b) 移動なし、(c) 当時のワークロード」
という前提つきである。現代では前提が崩れる場面が多い。

- **マルチスレッド**では、スレッドキャッシュやアリーナ分割が
  「方針の良さ」とは別の断片化（[blowup](multithread.md)）を生む。
- **ヒュージページ**を使うと、2 MiB 単位の中の小さな生存オブジェクトが
  ページ全体を居座らせる新種の断片化が起きる（[局所性の章](locality.md)）。
- **長時間稼働するサーバ**（数週間落ちないプロセス）では、
  1% の断片化でも積み重なって問題になり、実際に Google は
  自社の TCMalloc の断片化を倉庫規模で実測・分析する論文を
  2024 年に出している[](#cite:zhou2024)。

つまり「解決済み？」の答えは時代とともに「ふたたび未解決」へ揺り戻している。
これが第 III 部の残りの章につながる。

## 断片化を抑える設計上の手筋

理論と実測を踏まえると、実用アロケータが断片化対策として
採っている手筋が整理できる。

- **良い配置方針を使う**: アドレス順 first-fit／best-fit。
  next-fit を避ける[](#cite:johnstone1998)。
- **必ず併合する**: 境界タグで即時併合（[フリーリストの章](free-list.md)）。
  ただし速度のため一部は遅延併合し、定期的にまとめて併合する。
- **サイズクラスを細かすぎず粗すぎず**: 内部断片化 2 割程度を上限に
  （[サイズクラスの章](segregated-lists.md)）。
- **寿命の近いものを近くに置く**: 同時期に確保されたものは同時期に
  死にやすいという経験則。Barrett と Zorn は確保時点で寿命を予測して
  分離配置する手法を 1993 年に提案し[](#cite:barrett1993)、
  この発想は後の機械学習による割り当て[](#cite:maas2020)へ直結する。
- **空ページを OS に返す**: `madvise`（[OS とのインターフェースの章](os-interface.md)）。
  「在庫を持ちすぎない」ことも断片化（に見えるメモリ膨張）対策である。

> [!TIP]
> アプリ側でできる断片化対策もある。確保サイズを揃える、
> 寿命の違うオブジェクトを別のアロケータ／arena に分ける、
> 一括解放できる場面では region アロケータを使う、などだ。
> region 型が汎用 malloc より明確に有利という Berger らの測定
> [](#cite:berger2002)は、ここでも効いてくる。

## まとめ

断片化は内部（ブロック内の無駄）と外部（ブロック間の散らばり）に分かれ、
測定には定義の注意が要る。理論的には移動なしアロケータに
$\log n$ 倍の最悪下界があり[](#cite:robson1971)、
方針ごとの最悪順位は平均と逆転しうる[](#cite:robson1977)。
一方、実プログラムでは良い方針なら断片化はごく小さい[](#cite:johnstone1998)が、
マルチスレッド・ヒュージページ・長時間稼働という現代的条件で
問題は再燃している[](#cite:zhou2024)。
次章では、その「再燃」の最大要因——マルチスレッド——に踏み込む。
